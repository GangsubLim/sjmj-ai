# 품목 크롭 확장은 알고리즘으로, 완전 자동화는 로드맵 목표로 삼지 않는다

큐레이션 첫 실사례(job #22)에서 손글씨가 `ITEM_X`(`handwriting/rows.py:26`, 고정 폭 100~392px)
우측 경계를 넘어 "도리까이"가 "도"만 크롭되는 문제가 발견됐다. 이를 계기로 "품목명 크롭 엔진 자체를
학습시킬 수 있는가"를 검토했다.

## 결정

크롭 경계는 학습 가능한 모델이 아니라, 기존 잉크 검출 공식(`fewshot.py:tight_crop`·
`rows.py:_ink_mask`와 동일한 `(gray-blur) < -28` 국소대비 마스크)을 재사용해 **우측으로만**
동적으로 넓히는 알고리즘으로 개선한다. 좌측·상하 경계는 이번 범위에서 제외한다(상하 확장은
인접 행 획이 서로 겹치는 문제가 있어 더 까다롭다). 확장은 `AMOUNT_X`(612px) 앞에서 안전
여유를 두고 멈춘다. 잔여 오탐(예: "타이어 418 x 4"처럼 품목명과 수량이 gap 없이 붙어 쓰인
경우, "경유 1차/2차"처럼 품목칸을 비우고 수량 칸에 회차만 적는 경우)은 알고리즘으로 해소하지
않고 큐레이션(사람 배제)에 맡긴다.

## Considered Options

- **지도학습 기반 크롭 바운딩박스 검출 모델**: 기각. 현재 큐레이션은 정답 텍스트 라벨만
  캡처하고 정답 bbox는 어디에도 캡처되지 않아 학습 신호가 없다. 도입하려면 큐레이션 UI에
  bbox 교정 기능을 새로 만들어야 하는 훨씬 큰 범위의 작업이다.
- **품목명도 금액처럼 VLM으로 전체 전사**(크롭 경계 문제 자체를 회피): 기각. SP2 PoC에서
  이미 실측됨 — 범용 VLM 자유전사 baseline 13% vs 현재 작성자-특화 retrieval 방식 top-1 36%
  (`docs/superpowers/specs/2026-06-25-sp2-handwriting-recognizer-findings.md`). 크롭 경계
  문제를 없애려다 인식 정확도 자체가 후퇴한다.
- **"완전 자동화"(사람 개입 0)를 로드맵 목표로 명시**: 기각. "경유 1차/2차", "타이어 사이즈
  생략+합산" 같은 관행은 손글씨 이미지 자체에 전산화 규칙이 담겨있지 않아, 크롭·인식 품질과
  무관하게 사람 판단이 필요하다. 목표는 "재학습을 돌릴수록 큐레이션에서 사람이 배제·수정해야
  하는 비율이 줄어드는 것"으로 재정의한다.

## Consequences

- `training_pairs`에 정답 bbox를 캡처하는 인프라가 없다는 게 재확인됐으므로, 훗날 진짜
  학습 기반 검출기가 필요해지면 큐레이션 UI 확장(Phase B 이후)이 선행돼야 한다.
- 크롭 폭이 잡마다 가변적이 되므로, 임베딩 직전 `square()`(`fewshot.py:43`)의 기존
  `tight_crop` 안쪽 타이트닝은 그대로 유지된다 — 별도 변경 불필요.
