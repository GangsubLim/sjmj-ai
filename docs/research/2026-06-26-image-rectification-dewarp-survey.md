# 영수증 이미지 기하 보정(워프·정면화·dewarp) 조사

- 작성일: 2026-06-26
- 맥락: SP2 손글씨 인식 파이프라인의 이미지 정제(워프/crop/정규화) 단계 개선
- 상태: 조사 메모 (미커밋 유지)

---

## 0. 한 줄 결론

"포토샵/스마트폰처럼 격자에 맞춰 자동 보정"은 **단일 모델이 아니라 4개의 독립 문제(A/B/C/D)를 조합한 파이프라인**이다. SP2는 A(deskew)·B(원근)를 이미 OpenCV로 직접 처리 중이다.

샘플 실측(§4-bis) 결과 **종이 휨(D)은 "전면 구김"이 아니라 세로로 긴 영수증의 우측 가장자리 말림(curl)이 주류**이고, 하필 **금액 열**이 휜다. 그래서:

- **1차 해법은 풀 dewarp 모델이 아니라, 기존 `hline_ys()`(수평 파랑선) 인프라를 재사용한 행 단위 piecewise 휨 보정**이다 — 비용·리스크가 낮고 ROI(금액 열)에 정확히 작용하며, **렌즈 왜곡(C)으로 인한 행선 곡률까지 함께 흡수**한다.
- **C는 독립 단계로 두지 않는다** — EXIF·원본이 모두 없어 self-calib만 가능한데 종이 휨과 분리가 어렵고, 위 piecewise가 행선 곡률을 흡수하므로 실익이 낮다.
- **풀 dewarp 모델(UVDoc 등)은 piecewise로 안 펴지는 심한 구김에만 fallback**. 단 SP2는 고정 인쇄 양식이라 무조건 적용 시 인쇄 격자선을 망가뜨릴 수 있어 "휨 감지 + 격자 보존 가드레일" 하에서만 쓴다.

## 1. 문제 4분류

"기울어짐/왜곡 보정"은 자유도가 다른 4개 문제다. 각각 도구·난이도·SP2 발생 양상이 다르다.

| | 문제 | 정체 (자유도) | 평면 가정 | SP2 발생 | 현재 처리 |
|---|---|---|---|---|---|
| **A** | 기울어짐 보정 (deskew) | 평면 내 회전 (1 DOF) | O | 상시 | **함** — `rectify.py` Hough 수평 파랑선 각도 |
| **B** | 원근 복구 (정면화, rectification) | 평면 호모그래피 (8 DOF) | O | 상시 | **함** — `grid_v4.py warp()` 4코너→900×2100 |
| **C** | 렌즈 왜곡 (배럴/핀쿠션) | 카메라 방사·접선 왜곡 | △ | **간헐 (확인됨)** | 안 함 |
| **D** | 국소 비강체 변형 (종이 휨·구김) | 3D 곡면 펴기 (수천 DOF) | **X** | **간헐 (확인됨)** | 안 함 |

- **A·B는 평면 가정이 성립**하는 전역(global) 변환 → OpenCV 닫힌 해로 충분.
- **C는 카메라 고유 특성** → 캘리브레이션 또는 렌즈 프로파일.
- **D는 종이가 3D로 휘어 평면 가정이 깨진 경우** → 닫힌 해 없음, 딥러닝 dewarp 모델 영역. 사용자가 말한 *"내부 왜곡된 부분만 부분적으로 펴주는"* 이 정확히 D다.

핵심: **A·B로는 D를 못 고친다.** 4코너 워프(B)는 종이를 "평평한 사각형"으로 가정하므로, 종이가 휘어 있으면 코너만 맞고 안쪽 행은 여전히 휜다.

## 2. 포토샵 / 스마트폰이 실제로 하는 일

화려해 보이지만 내부는 위 4단계의 조합이다.

- **스마트폰 문서 스캐너** (Adobe Scan, MS Lens, Apple/Google 노트 스캔)
  - 코너 검출(문서 segmentation) → **B(4점 원근보정)** + 약한 **A(deskew)**. = SP2가 이미 하는 파이프라인.
  - 고급 앱만 옵션으로 **D(곡면 dewarp)**를 추가. 기본 동작 아님.
  - 코너 검출에 색·에지 휴리스틱이 아니라 **학습 기반 문서 세그멘테이션(U²-Net 류)**을 쓰는 점이 SP2 색분리 방식과의 차이.
- **Lightroom/Photoshop**: "Upright"/"Lens Correction" = **A+B**(소실선 정렬) + **C**(EXIF 렌즈 프로파일 기반 undistort). 격자 자동 정렬은 vanishing line 검출.

→ "통째로 가져다 쓸 단일 모델/SDK"는 없다. 각 단계의 오픈소스를 조합한다.

## 3. 도구 카탈로그

### 3.1 A·B — 전역 변환 (이미 사용 중, 보강용)

| 도구 | 용도 | 비고 |
|---|---|---|
| `cv2.getPerspectiveTransform` + `warpPerspective` | B 원근보정 | SP2 `grid_v4.py`에서 사용 중 |
| `cv2.HoughLines` / PCA / `jdeskew` | A deskew | SP2 `rectify.py` Hough 사용 중 |
| 학습 기반 문서 세그멘테이션 (U²-Net, PaddleSeg "deep document scanner") | 코너/경계 검출 강건화 | 색분리가 놓치는 **옅은 하단 경계**를 형태로 검출 → §7 세로 완전성 문제에 직접 도움 |

### 3.2 C — 렌즈 왜곡 보정

- `cv2.undistort` + 카메라 캘리브레이션(체스보드로 내부 파라미터 K, 왜곡계수 D 추정).
- **스마트폰은 EXIF에 기기/렌즈 정보**가 있어 lensfun 등 렌즈 프로파일 DB로 추정 가능. 단 **본 데이터셋(`Resized_*`)은 리사이즈로 EXIF가 소실**돼 이 경로 불가(§4-bis-4).
- 적용 위치: **B(4점 워프) 이전**에 한 번. (워프 후엔 직선성이 깨져 보정 기준이 사라짐)
- SP2 영향도: 영수증이 프레임 중앙을 크게 차지하면 방사 왜곡 영향이 작다. **가장자리에 걸치거나 광각·근접 촬영 시** 격자선이 배럴로 휘어 두드러진다.

### 3.3 D — 국소 비강체 dewarp (딥러닝 local model)

활성 repo·pretrained weight 공개 기준. 입력은 단일 RGB 이미지, 출력은 펴진 이미지(또는 backward map).

| 모델 | 연도/출처 | 특징 | 비고 |
|---|---|---|---|
| **UVDoc** | SIGGRAPH Asia 2023, ETH Zürich | 3D grid mesh 예측, 경량·빠름, 자체 데이터셋 공개 | `igl.ethz.ch/projects/uvdoc` |
| **DocTr++** | 2023 | geometry(펴기) + illumination(그림자 제거) 동시, 실전 완성도 높음 | weight 공개 |
| **DocScanner** | 2023 | 반복 정제 구조, 경량 | |
| **DewarpNet** | 2019 (원조) | Doc3D 합성 데이터셋 기반 | 후속 연구의 베이스라인 |
| GeoTr / DocGeoNet / Marior / DDCP | 2021~2022 | 각종 변형 | |
| (최신) Diffusion Transformer rectification | CVPR 2025 W | 확산 기반, 무거움 | 연구 단계 |

- 종합 모음집(라이선스·weight 일괄 확인): **`github.com/ZZZHANG-jx/Recommendations-Document-Image-Processing`**
- ⚠️ **라이선스**: 다수가 연구용/비상업 제약. 상업 투입 전 개별 확인 필수.
- ⚠️ **학습 도메인**: 전부 **영수증·책장처럼 물리적으로 휜 자연 문서**로 학습됨. 인쇄 격자가 촘촘한 정형 양식은 학습 분포 밖.

## 4. SP2 도메인 적용 시 핵심 주의 (인쇄 양식 특수성)

D 모델을 무비판 적용하면 안 되는 이유:

1. **격자선 오인 위험** — dewarp 모델은 텍스트 라인·테두리의 곡률을 "휨 단서"로 삼는다. SP2 양식의 **인쇄된 표 격자·청색 라인이 강한 직선 신호**라, 종이가 평평한데도 모델이 이를 왜곡으로 오해해 **멀쩡한 직선을 비틀** 수 있다.
2. **OCR 민감도** — 손글씨 숫자 인식은 미세 기하 왜곡에 민감하다. 불필요한 dewarp가 인식률을 **깎는** 역효과.
3. **현행 "고정 피치 행그리드(P≈81px)"는 도메인 특화 dewarp**다. 양식 구조(행 간격)를 아는 만큼, 구조를 모르는 일반 모델보다 정확하고 통제 가능하다.

→ 따라서 D는 **"항상 적용"이 아니라 "휨이 감지된 프레임에만, 그리고 인쇄 격자 보존을 깨지 않는 안전장치와 함께"** 적용해야 한다.

## 4-bis. 실측 관찰 (샘플 4종)

`data/image/` 의 사용자 지정 4개 사례를 직접 확인한 결과.

| 샘플 | 회전(A) | 원근(B) | 종이 휨(D) | 비고 |
|---|---|---|---|---|
| `Resized_20250822_114723` | 경미(반시계) | 약 | **경미** — 하단 세로선 미세 곡선 | 대체로 양호 |
| `Resized_20250825_153457` | 뚜렷(시계) | 중 | **뚜렷** — **우측 가장자리 말림(curl)**, 행선이 위로 휨 | 금액 열이 휨 영역 |
| `Resized_20250901_182016` | 경미 | 약 | 거의 없음 | 양호 케이스 |
| `Resized_20250913_130056` | 경미 | 중 | **뚜렷** — 우측 말림 + 주름, 배경에 타 장부 노출 | 코너검출 배경 혼동 위험 |

### 관찰에서 나온 핵심 사실

1. **휨 형태는 "전면 구김"이 아니라 "세로로 긴 영수증의 한쪽(주로 우측) 가장자리 말림(curl)"이 주류.** 사용자가 말한 "간혹·살짝"과 일치. → 풀 3D dewarp(UVDoc/DocTr++)까지 안 가도 **행 단위 piecewise 보정**으로 상당 부분 펴질 가능성이 높다(§5 참조).
2. **하필 휘는 곳이 맨 오른쪽 "공급대가(금액)" 열이다.** SP2에서 가장 중요한 숫자가 있는 영역이 휘므로, 휨 보정의 ROI는 명확히 우측 금액 열이다. (`grid_v4.py` 공급가 열 추출과 직결)
3. **4점 워프(B)로는 이 curl을 못 편다** — 코너 4점은 맞지만 안쪽 행이 휘어, 금액 셀 경계가 어긋난다. B 이후의 보정 단계가 필요함을 실측이 확인.
4. **렌즈 왜곡(C) 프로파일 보정은 불가 → self-calibration으로 확정** — `Resized_*` 는 **EXIF가 전부 소실**(Make/Model/Focal=None, 960×1280)이고 **리사이즈 전 원본도 없음(확인됨)**. 따라서 lensfun/EXIF 프로파일도, 기기 특정 체스보드 캘리브레이션도 불가(어느 기기로 찍었는지조차 모름). **남은 유일한 경로는 self-calibration** — 인쇄된 표 격자(직선)가 직선이 되도록 왜곡계수(division model 단일 λ 또는 OpenCV k1,k2)를 영상에서 역추정(plumb-line 방식).
5. **배경 혼동 위험**(`913`): 배경에 유사 인쇄 장부가 노출 → 색분리는 영수증 청색 격자를 잡지만, 배경도 비슷하면 코너검출이 흔들린다. §5의 문서 세그멘테이션 보강 근거를 강화.

## 5. 권고 (의사결정 가이드)

빈도("간혹")·정도("살짝")와 **실측(§4-bis: 우측 가장자리 curl이 주류)**을 고려한 단계적 접근. 위로 갈수록 우선순위 높음.

1. **[High] 코너/경계 검출 강건화 (A·B 앞단)** — 학습 기반 문서 세그멘테이션을 색분리 코너 검출의 **폴백/보강**으로. §7 "하단 옅은 부분 행 누락(silent loss)" + `913`류 **배경 장부 혼동**에 직접 대응. D 모델보다 적은 리스크로 즉시 실익.
2. **[High] 행 단위 piecewise 휨 보정 (D의 경량 해법, 풀 모델보다 우선)** — 실측상 휨이 "우측 가장자리 curl"이므로 전면 3D 펴기까지 갈 필요가 적다. **기존 `grid_v4.hline_ys()`(수평 파랑선 검출) 인프라를 재사용**해 각 행의 좌·우 끝 y를 따라가며 **행별 affine/스플라인으로 곡선 행선을 수평으로 펴는** 방식. 비용·리스크 낮고 ROI(우측 금액 열)에 정확히 작용. **이 도메인에선 이게 1차 권장.**
3. **[Low/선택] 렌즈 왜곡(C) 독립 보정** — EXIF·원본 모두 없으므로 **self-calibration**이 유일 경로(§4-bis-4). 그러나 **실익이 제한적**이다:
   - **2번 piecewise 행 보정이 렌즈로 인한 행선 곡률까지 함께 편다.** SP2는 행(가로) 단위로 숫자를 읽으므로 행선만 수평이면 충분 → C를 별도 `undistort` 단계로 분리할 이유가 약하다.
   - 게다가 self-calib는 "전 영상에 일관된 방사 왜곡" 가정인데, **같은 영상에 종이 휨(D)이 섞여 직선이 휘어** 렌즈 성분과 분리가 어렵다. 분리하려면 *같은 기기로 찍은 여러 장*을 모아 추정해야 하나(장마다 다른 종이 휨은 평균 상쇄, 공통 렌즈 성분만 잔류), 과거 사진은 **기기 불명·혼재**라 이 전제가 깨진다.
   - 결론: C는 독립 단계로 넣지 말고, **세로선 직선성까지 엄밀히 필요해질 때만** self-calib를 검토. 우선은 2번이 흡수.
4. **[Cond] 풀 곡면 dewarp(D) 모델** — 2번 piecewise로 부족한 **심한 구김** 프레임에만 fallback. "휨 감지" 후에만 적용:
   - 감지 신호: B(4점 워프) 후에도 행선 직선성 잔차가 임계 초과 / 행 피치가 위치별로 흔들림.
   - 가드레일: dewarp 전후 **인쇄 격자 직선성**을 비교해 오히려 나빠지면 원본 채택.
   - 모델: **UVDoc(경량) PoC** → 효과 시 DocTr++(조명 보정 겸용) 비교.

### 적용 순서(파이프라인 위치)

```
원본
 → 코너/경계 검출 (색분리 + 학습 세그멘테이션 폴백, 배경 혼동 방지)
 → [B] 4점 원근보정 (정면화)
 → [D-경량] 행선 추종 piecewise 휨 보정 (hline_ys 재사용)  ← 1차. 렌즈 곡률도 함께 흡수
 → [심한 구김 감지?] ──예──> [D-풀] UVDoc dewarp (격자 보존 가드레일)  ← fallback
 → [A] deskew (Hough) + 고정 피치 행그리드
 → 셀 crop → OCR

 ※ [C] 렌즈 self-calib는 별도 단계로 넣지 않음 — D-경량이 행선 곡률을 흡수.
   세로선 직선성까지 엄밀히 필요해질 때만 코너검출 전에 1회 삽입.
```

## 6. 향후 데이터 수집 권고 (운영)

과거 데이터는 손 못 대지만, **신규 촬영분은 저비용으로 보정 여지를 크게 넓힐 수 있다.**

- **리사이즈 전 원본 + EXIF 보존.** `Resized_*`로 덮어쓰지 말고 원본을 별도 보관. EXIF가 살아 있으면 렌즈 프로파일(lensfun) 기반 C 보정이 즉시 가능해진다.
- **가능하면 동일 기기로 촬영.** 기기가 한정되면 1회 체스보드 캘리브레이션으로 정확한 왜곡계수를 확보할 수 있고, self-calib도 안정된다.
- 위 두 가지만 지키면 C는 "불가/불안정"에서 "프로파일 1회로 해결"로 바뀐다.
