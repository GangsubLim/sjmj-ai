# 품목 크롭 우측 확장 설계

큐레이션 첫 실사례(job #22)에서 손글씨가 품목칸 고정폭 경계를 넘어가 정답 라벨의 일부만
크롭되는 문제가 발견됐다. 크롭 경계를 잉크 프로파일 기반으로 우측으로만 동적 확장하는
알고리즘을 도입하고, 로컬 PoC 사진으로 검증한다.

## 1. 배경과 문제

### 1.1 현황 (코드 확인)

- 품목 크롭은 `handwriting/rows.py:26`의 고정 상수 `ITEM_X = (100, 392)`를 그대로 슬라이싱해서
  만든다(`infer_photo.py:159`: `w[r.box[0]:r.box[1], x1-4:x2+4]`). 학습 가능한 파라미터가
  없는 순수 기하학적 처리다.
- 이 `crops[i]` 배열은 라이브 추론의 품목 임베딩 입력(`embed_crops`)과 큐레이션·학습쌍
  저장용 `row-{i}.png`(`infer_job.py:78`) **양쪽에 동일하게** 쓰인다.
- 임베딩 직전 `square()`(`fewshot.py:43`)가 `tight_crop()`으로 잉크 bbox 안쪽 타이트닝을
  하지만, 이건 이미 잘려서 사라진 픽셀은 복구하지 못한다.

### 1.2 실사례 (job #22)

- 손글씨가 품목칸 우측 경계(392px)를 넘어 "도리까이"가 "도"만 크롭됨.
- 사용자 관찰: 품목명은 (a) 시작점 자체가 우측으로 들여써지거나, (b) 길이가 길어 우측으로
  넘치거나, (c) 위/아래 행 경계를 넘어가는 경우가 있음. 금액칸은 상대적으로 안정적.

### 1.3 도메인 사실 (이번 grilling에서 확정, ADR 0005 참조)

- 크롭 경계 자체는 재학습(품목 인식기 fine-tune) 대상이 아니다 — `CONTEXT.md` "품목 크롭"
  용어 참조.
- "완전 자동화"는 로드맵 목표가 아니다. "경유 1차/2차"(품목칸을 비우고 수량칸에 회차만
  기록), "타이어 사이즈 생략+합산" 같은 관행은 손글씨 이미지 자체에 전산화 규칙이 없어
  크롭·인식 품질과 무관하게 사람 판단이 필요하다.
- `ITEM_X` 우측(392)과 `AMOUNT_X` 좌측(612) 사이 220px는 "규격/수량/단가" 3개 컬럼
  공간이며, 현재 파이프라인은 이 3칸을 쓰지 않는다. 다만 실제로 수량이 여기 적히는 경우가
  간혹 있어(경유 1차/2차 등), 무조건 "가장 우측 잉크"를 취하면 안 되고 gap 기반으로 멈춰야
  한다.

## 2. 핵심 결정

| 결정      | 내용                                                                                                                                                                                                                      |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 확장 방향 | 우측만. 좌/상/하 확장은 비범위(상하는 인접 행 획 겹침 문제로 더 까다로움)                                                                                                                                                 |
| 알고리즘  | 잉크 프로파일(`_ink_mask`와 동일한 `(gray-blur)<-28`, **인쇄 세로선 억제 후**) 기반 gap 정지, `AMOUNT_X` 앞 안전 여유 상한. 결과는 항상 `ITEM_X[1]` 이상으로 클램프 — **확장 전용, 절대 좁아지지 않음**                   |
| 잔여 오탐 | 알고리즘으로 해소하지 않음(예: "418 x 4" gap-less 케이스) — 큐레이션 배제가 안전망                                                                                                                                        |
| 적용 범위 | `extract_rows_for_job` 한 경로 수정 → 라이브 추론 임베딩 + **앞으로 저장되는** `row-{i}.png` 학습쌍 + 데모 오버레이. 뱅크 부트스트랩 재생성(`label_inspect.build_rows`·`dataset_build`)은 별개 고정폭 경로라 미포함(§3.3) |
| 검증 방법 | 로컬(MacBook) 기존 PoC 사진으로 기존/신규 크롭 비교 HTML(오버레이 확장) 눈으로 확인                                                                                                                                       |
| 비범위    | bbox 학습 검출 모델, 완전 자동화, 운영 job 백필 메커니즘(로컬 검증 후 별도 결정)                                                                                                                                          |

## 3. 설계 상세

### 3.1 신규 함수 (`handwriting/rows.py`)

```python
def item_crop_right_bound(warp, row_band, item_x=ITEM_X, amount_x=AMOUNT_X,
                           gap_px=..., margin_px=...) -> int:
    """품목칸 잉크 프로파일에서 우측 경계를 동적으로 구한다(확장 전용).

    세로 잉크 프로파일을 만들되, 인쇄 세로선(칸 경계선)은 먼저 억제한다
    (`_remove_hlines`의 세로 버전 — 근거 §3.4). item_x[0]부터 오른쪽으로 훑어
    **첫 잉크가 나온 뒤부터** gap_px 이상 잉크가 끊기는 지점, 또는
    amount_x[0] - margin_px 중 먼저 오는 지점을 후보로 잡는다.

    불변식: 반환값은 항상 max(후보, item_x[1]). 기존 고정폭보다 절대 좁아지지
    않는다(확장 전용). 좌측 들여쓰기로 시작점이 item_x[0]보다 우측이어도,
    선행 공백은 정지 gap으로 치지 않는다(첫 잉크 이후에만 gap을 센다).
    잉크가 전혀 없으면 item_x[1] 반환(행 동작 불변).
    """
```

기존 `stroke_profile_col`(Y축 행 분할용, 현재 미사용 보류 유틸)와 같은 프로파일+세그먼트
뼈대를 X축에 재적용한다. `gap_px`/`margin_px`의 정확한 값은 §4 로컬 검증에서 실측 튜닝한다
(초기값은 정상 글자 간 간격보다 뚜렷이 큰 값 + `AMOUNT_X[0]`까지 최소 여유폭으로 잡는다).

**확장 전용 불변식(Blocker 리뷰 반영):** 손글씨 시작점이 품목칸 안에서 우측으로
들여써지는 실사례(§1.2 case a)가 있어, `item_x[0]`부터 순진하게 gap을 세면 선행
공백을 정지점으로 오인해 `item_x[1]`보다 좁은 값을 낼 수 있다. 이를 두 겹으로 막는다:
(1) gap 카운트는 첫 잉크 관측 이후에만 시작, (2) 최종 반환은 `max(후보, item_x[1])`로
클램프. 어떤 경우에도 크롭이 현재 고정폭보다 좁아지지 않음을 §5 단위테스트로 못박는다.

### 3.2 호출부 변경 (`handwriting/infer_photo.py:157-159`)

```python
x1, x2 = ITEM_X
ax0, ax1 = AMOUNT_X
crops = [
    w[r.box[0]:r.box[1], x1 - 4 : item_crop_right_bound(w, r.box) + 4]
    for r in news
]
```

`extract_rows_for_job`는 `infer_job`(운영)과 `process_one`(데모)이 공유하는 단일 경로이므로,
이 한 곳의 수정이 라이브 품목 임베딩·`row-{i}.png` 학습쌍 저장·데모 오버레이 세 곳 모두에
자동 반영된다. `square()`의 기존 `tight_crop` 안쪽 타이트닝은 변경 없이 그대로 유지된다.

### 3.3 적용 범위 밖 — 뱅크 부트스트랩 crop 경로 (High 리뷰 반영)

§3.2 수정은 **라이브 추론 query crop과 앞으로 저장되는 학습쌍 `row-{i}.png`만** 바꾼다.
재학습이 대조하는 뱅크(`bank.npz`)의 부트스트랩 crop은 별개 경로에서 여전히 고정폭으로
만들어지므로 이번 수정이 닿지 않는다:

- `handwriting/train_contrastive.py:94` `prepare()` → `label_inspect.build_rows()`
  (`report/sp2_spike/item/label_inspect.py:94`, `std=w[a:b, x1-4:x2+4]`)
- `handwriting/dataset_build.py:319` (`crop=warps[cn][a:b, x1-4:x2+4]`)

두 문제가 있다. (1) **crop 폭 분포 어긋남**: 이번 수정만으로는 뱅크는 좁은 crop, 라이브
query는 넓은 crop이 된다. `square()`의 `tight_crop`이 잉크 bbox로 조이므로 넓은 crop이 실제
담는 잉크 내용 자체가 달라져 임베딩이 이동한다 — 재학습으로 확장의 이점을 온전히 얻으려면
이 부트스트랩 경로도 같은 확장 로직으로 재생성해야 한다. (2) **캐시 자동 무효화 안 됨**:
`prepare()`의 clean crop 캐시 키는 `md5("v2" + corr)`로 **crop 기하를 반영하지 않아**, 확장
로직을 넣어도 캐시를 수동 삭제하지 않으면 옛 좁은 crop이 그대로 재사용된다.

이 정합화(부트스트랩 경로 확장 적용 + 캐시 무효화 + 뱅크 재생성)는 §7의 운영 백필 결정과
성격이 같다(로컬 검증 통과 후 별도 결정). **이번 스펙 비범위**로 명시한다.

### 3.4 인쇄 세로선 억제 (Medium 리뷰 반영)

SP2 `label_inspect.py:8-12`가 이미 기록한 사실: overflow 자동 기하검출을 포기한 이유가
"손글씨가 인쇄 경계선·이웃칸과 분리 불가능한 위치에 있어, 경계선이 stroke로 잡혀 품목·이웃칸을
잇는다"는 것이다. 즉 `ITEM_X[1]`(392) 부근이나 컬럼 사이의 **인쇄 세로 칸선**이 잉크로 잡히면
프로파일이 끊기지 않아 gap-stop이 무력화되고, 크롭이 `AMOUNT_X` 앞 상한까지 통째로 늘어난다.

현재 `rows.py`엔 `_remove_hlines`(전폭 수평선 제거)만 있고 세로선 대응이 없다. 따라서 세로
프로파일을 만들기 전에 **세로 방향 억제**를 넣는다 — `_remove_hlines`를 전치(transpose)로
재사용하거나 세로 대응 헬퍼를 신설한다(행 높이에 걸쳐 연속하는 얇은 세로 성분 제거). 이는
gap-stop 신뢰성의 선결 조건이며, 순수 numpy 합성 테스트(§5)로 세로선·옅은 컬럼선 케이스를
반드시 커버한다.

## 4. 로컬 검증 도구

`process_one`의 오버레이 HTML 생성 경로를 확장해, 같은 사진에서 "기존 고정폭 크롭"과
"신규 확장 크롭"을 나란히 보여주는 비교 뷰를 추가한다. 출력은 `report/`(gitignore) 하위 —
기존 관례와 동일하게 로컬 전용이며 pytest 대상이 아니다. 이 도구로 로컬 PoC 사진 중
"도리까이"류(우측 넘침), "경유 1차/2차"류(gap 있는 별개 토큰), "타이어 418 x 4"류
(gap 없는 붙어쓰기) 패턴을 눈으로 확인한다.

## 5. 테스트

`tests/`에 합성 numpy 배열 기반 순수함수 단위테스트(ml/ 관례상 실 이미지 비의존):

- 품목명이 `item_x` 안에 다 들어가는 정상 케이스 → 경계 불변
- 우측으로 넘치는 케이스 → 경계가 잉크 끝까지 확장
- **좌측 들여쓰기 + 우측 overflow** 케이스(시작점이 `item_x[0]`보다 우측, 끝은 `item_x[1]`
  초과) → 선행 공백을 gap으로 오인하지 않고, 반환이 `item_x[1]` 이상이며 잉크 끝까지 확장
  (Blocker 리뷰 — 확장 불변식 회귀 방지)
- **좁아지지 않음 불변식**: 어떤 입력에도 반환 `>= item_x[1]` (클램프 검증)
- 넓은 gap 뒤 별개 토큰이 있는 케이스 → gap에서 정지, 확장 안 됨
- **인쇄 세로 칸선이 프로파일에 걸린 케이스**(`item_x[1]` 부근 얇은 세로선) → 세로선 억제
  후 gap-stop이 정상 동작, 세로선을 잉크로 오인해 상한까지 늘어나지 않음 (Medium 리뷰)
- **옅은 세로 컬럼선** 케이스 → 억제되어 경계에 영향 없음
- gap 없이 붙어있는 케이스 → 알려진 한계로 확장됨을 명시적으로 assert(테스트 주석에
  "큐레이션이 걸러야 하는 케이스"임을 남긴다)
- 잉크가 `amount_x` 근처까지 이어지는 케이스 → `margin_px` 상한 적용
- 잉크가 전혀 없는 행 → `item_x[1]` 그대로 반환

## 6. 에러 처리 / 엣지 케이스

- 잉크가 전혀 없는 행(empty 타입) → 원래 `item_x[1]` 반환(동작 불변)
- 좌측 시작점 들여쓰기 → **좌측 경계**(`x1-4`)는 이번 스펙 비범위로 불변. 단 **우측 경계
  스캔**에는 영향을 준다: 선행 공백을 첫 잉크 전까지 gap으로 세지 않고, 최종값을
  `item_x[1]` 이상으로 클램프해 좁아짐을 막는다(§3.1 불변식)
- 인쇄 세로 칸선이 잉크로 잡히는 경우 → §3.4 세로선 억제로 선처리 후 프로파일링

## 7. 비범위 (YAGNI)

- 상/하 경계 확장(인접 행 획 겹침 문제, 더 까다로움 — 필요성이 실측되면 별도 스펙)
- bbox 학습 기반 검출 모델(ADR 0005에서 기각 — 정답 bbox 캡처 인프라 자체가 없음)
- "완전 자동화"(ADR 0005에서 로드맵 목표 아님으로 확정)
- 운영 job(#22 포함) 백필 메커니즘 — 전용 백필 스크립트 vs 큐 재사용 여부는 이 스펙의
  알고리즘이 로컬 검증을 통과한 뒤 별도로 결정한다. 그때까지 job #22는 첫 학습쌍만
  excluded 처리하고 검수 완료(`curation_reviewed`)는 보류한다.
