# SJMJ OCR 통합 파이프라인 — macmini 이전 & 리팩토링 개괄 (검토용)

- 작성일: 2026-06-24
- 성격: **다음 세션 검토용 개괄(overview)** — 결정 문서 아님. 논점·옵션·열린 질문 정리가 목적.
- **새 프로젝트명: `sjmj-ai`** (확정) — "SJMJ 업무 전반을 AI로 자동화하는 플랫폼". invoice OCR이 **첫 모듈**이고, 이후 다른 업무 자동화 모듈로 확장.
- 관련: [`2026-06-24-invoice-htr-tech-review.md`](./2026-06-24-invoice-htr-tech-review.md), [`2026-06-24-invoice-handwriting-ocr-poc-design.md`](./2026-06-24-invoice-handwriting-ocr-poc-design.md)

## 1. 왜 옮기나 (문제)

현재 SJMJ-Web은 **닷홈(공유 호스팅, Apache+PHP)** 배포 구조다. OCR 자동화 파이프라인의 요구사항과 충돌한다:

| OCR 파이프라인 요구 | 닷홈(공유 호스팅) 제약 |
|---------------------|------------------------|
| ML 추론(PaddleOCR/YOLO) 상시 구동 | Python·GPU/MPS 불가, 장기 프로세스 불가 |
| 모델 fine-tune(학습) | 연산 자원·라이브러리 없음 |
| 비동기 작업 큐(추론·재학습 잡) | 백그라운드 워커 불가 |
| 피드백 루프(교정→재학습) 자동화 | 상태 유지·스케줄러 불가 |
| 대용량 이미지·모델 파일 | 파일시스템·메모리 제약 |

→ **자체 서버(macmini, M5 Max/128GB)로 옮기고, DB+백엔드+프론트+ML을 통합 파이프라인으로 재구성**하는 것이 전제. macmini는 on-prem VLM·OCR 추론·학습을 모두 감당할 수 있음(기술검토 §5b 실측 근거).

## 2. 목표 구조 (한눈에)

```
[현재]  브라우저 ─ 닷홈(Apache) ─ PHP 백엔드 ─ MySQL          (ML 없음)

[목표]  macmini (Docker compose, self-hosted)
        ┌─────────────────────────────────────────────┐
        │ frontend(React) ─ backend(API) ─ MySQL        │
        │        │                │                      │
        │        └─ 검수 UI       └─ ML 서비스(Python)   │
        │                            검출→인식→검산       │
        │                            ↕ 학습 파이프라인     │
        │                          worker(비동기 큐)      │
        └─────────────────────────────────────────────┘
```

## 3. 현재 자산 규모 (재작성 비용 가늠)

| 영역 | 규모 | 재사용성 |
|------|------|----------|
| 백엔드 PHP (4-layer, 30 routes, 226 테스트) | 60파일 / ~3,060 LOC | **작음** — 재작성 부담 낮음 |
| 프론트 React (TS/TSX) | 133파일 / ~14,200 LOC | **큼** — 재사용 가치 높음(유지 권장) |
| DB MySQL (8 테이블, 운영 400건+) | 스키마+데이터 | 그대로 이전 |
| 인프라 | `docker-compose.yml` + Makefile 이미 존재 | macmini 이전에 유리 |

## 4. 핵심 결정거리 (다음 세션 논의)

대부분 방향이 명확하나, **백엔드 언어** 하나가 진짜 결정 포인트다.

| 항목 | 방향 | 비고 |
|------|------|------|
| 인프라 | macmini + Docker compose 모노레포 | 거의 확정 |
| 프론트엔드 | **React 유지** | 14,200 LOC 재사용, API 클라이언트만 조정 |
| DB | **MySQL 유지** + 운영 데이터 이전 | 변경 없음 |
| ML 스택 | **Python** (PaddleOCR·YOLO·PyTorch/MLX) | 선택 여지 없음 |
| **백엔드 언어** | **❓ Python(FastAPI) 전환 vs PHP 유지+Python ML 분리** | **핵심 결정** |

### 백엔드 언어 — 두 안의 트레이드오프

- **안 ① Python(FastAPI)로 통합** — ML과 백엔드가 **단일 언어/스택**. 피드백 루프·추론 호출이 같은 런타임 안에서 자연스러움. 백엔드 3,060 LOC 재작성(작아서 부담 낮음). 226 PHP 테스트는 폐기·재작성. *장기 단순성 ↑, 초기 재작성 비용.*
- **안 ② PHP 백엔드 유지 + Python ML 마이크로서비스 분리** — 검증된 PHP 백엔드 보존, ML만 별도 서비스(REST/큐로 연동). **두 언어 운영**(배포·디버깅 복잡), 언어 경계마다 통합 비용. *초기 비용 ↓, 장기 복잡성 ↑.*

> 1인 운영·ML 중심·백엔드 소규모(3,060 LOC)를 감안하면 **안 ①(Python 통합)이 장기적으로 단순**하다는 게 현재 기울기. 단 최종 결정은 다음 세션에서.

## 5. 피드백 자동화 루프 (이 프로젝트의 핵심 신규 요소)

```
사진 업로드 → [큐] → ML 추론(검출→인식→산술검산) → 초안 JSON
   → 검수 UI에서 사람 교정 → 운영DB 확정 저장
   → (산술검산 통과분 + 사람 교정분)이 곧 학습 정답 → 자동 축적
   → 임계 도달 시 재학습 트리거 → 모델 갱신 → 다음 추론 정확도 ↑
```

- 운영DB가 이미 **정답지(400건+, 계속 축적)** 이므로(0단계 감사), 사진↔DB 자동 매칭(`grand_total`+`issue_date`)으로 **학습 데이터가 운영 중 저절로 쌓이는** 구조가 가능. 이게 닷홈에선 불가능했던 핵심.

## 6. 새 레포 구조 — `sjmj-ai` (모노레포)

> invoice OCR은 **첫 모듈**. 향후 다른 업무 자동화 모듈을 같은 레포에 확장하는 구조.

```
sjmj-ai/  (새 레포, 모노레포)
├── apps/
│   └── invoice-ocr/      # 첫 모듈: 수기 거래명세서 OCR 자동입력
│       ├── frontend/     # React (기존 SJMJ-Web 이식 + 검수 UI)
│       ├── backend/      # API (FastAPI 안① / PHP 안②)
│       └── ml/           # 검출→인식→정규화→검산 + 학습(사진↔DB 자동매칭)
├── packages/             # 모듈 간 공통(UI·타입·DB 클라이언트 등)
├── worker/               # 비동기 큐(추론·재학습 잡)
├── db/                   # MySQL 스키마·마이그레이션 (기존 이전)
├── docker-compose.yml
└── docs/
```

(※ 모노레포 vs 단일앱 구조 자체는 다음 세션 결정. 위는 "확장 비전"을 반영한 제안.)

## 7. 마이그레이션 & 리스크 (개괄)

- **데이터·도메인 이전**: 운영 MySQL 덤프 이전, 도메인 macmini로 전환(reverse proxy + HTTPS).
- **macmini 운영 리스크**: 단일 장애점(전원·네트워크·가용성) → 백업·모니터링·자동재시작 필요. (※ 별도 프로젝트에서 macmini 배포 운영 경험 있음)
- **전환 방식**: 닷홈 병행 운영하며 점진 이전 vs 새 레포 클린 시작 후 컷오버 — 다음 세션 결정.
- **보안**: 외부 노출 시 인증·방화벽. 민감 데이터(거래처·차량번호)는 on-prem 유지로 오히려 유리.

## 8. 다음 세션 검토 안건 (열린 질문)

1. **백엔드 언어**: Python(FastAPI) 통합 vs PHP 유지+ML 분리 — §4 트레이드오프 결정
2. **시작 방식**: 기존 레포 점진 리팩토링 vs **새 레포 클린 시작**(사용자가 새 폴더·레포 언급) + 자산 이식
3. **우선순위**: 인프라 이전(macmini 셋업) 먼저 vs OCR PoC(0~1단계) 먼저 vs 병행
4. **macmini 운영 준비**: 도메인·HTTPS·상시구동·백업 체계 현황
5. **범위 경계**: 이번 리팩토링에 OCR 파이프라인 어디까지 포함(추론만 vs 학습 루프까지)

---

> **요약**: 인프라(macmini+Docker), 프론트(React 유지), DB(MySQL 유지)는 방향이 분명하다. 진짜 결정은 **백엔드 언어(Python 통합 기울기)**와 **새 레포 클린 시작 여부**다. 다음 세션에서 이 두 가지를 정하고 본격 설계(brainstorming → plan)로 들어가면 된다.
