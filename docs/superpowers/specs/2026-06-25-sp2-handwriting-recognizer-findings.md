## 0. TL;DR

1. **로컬 VLM이 숫자 병목을 깼다** — M5 Max MLX로 **Qwen3-VL-8B**가 공급가 열 36장에서 DB 대비 **end-to-end recall 84.8%** vs SP1 stock PP-OCRv5 **~10%**. 외부 API 0, 0.6초/장. ⚠️ 84.8%는 (검출 정렬+grouping+인식)이 혼입된 set-recall **신뢰 하한**이지 isolated 인식 정확도가 아니다 — 정렬된 strip은 만점 다수라 인식 천장은 더 높다(§1). 측정 무결성은 §0-9·§8-A에서 선결.
2. **검출 = 연파랑 격자 색분리 + 템플릿 원근보정**. "격자=연파랑(고정)·손글씨=검정(가변)" **색 불변식**으로 SP1이 접은 괘선 추출을 돌파. 공급가 열 strip 안정 추출.
3. **GT 의미론 확정** — references→DB 정답은 **항목별 합산 공급가**. 손글씨↔DB 차이의 정체는 **약식 분해(grouping)**: 품목칸이 빈 행의 금액은 **바로 위 품목에 합산**.
4. **품목 직접 OCR은 실패(검증 완료)** — Qwen3-VL-8B **12.3%**(숫자 84.8%와 정반대). 원인은 개인 속기체 + DB 정식명 불일치 → 모델 키워도 안 풀리는 본질적 가독성. (§4)
5. **방향 = 작성자-특화 닫힌어휘 인식** — "전사"가 아니라 "글씨 패턴 → DB 마스터 386 품목 분류/검색". few-shot exemplar 최근접 → 표본 쌓이면 파인튜닝. 인식→DB매칭→저신뢰/신규는 사람등록→재학습 루프. (§5)
6. **데이터 계획** — 손글씨 표본 102/386은 38장만 이관된 탓. 사진 최대한 이관 시 마스터 대부분에 실표본 → 파인튜닝 연료. (§5)
7. **금액은 OCR만** — 단가 변동으로 DB 단가 금액 역산 보류. DB는 grouping 합검증 앵커로만. (§6)
8. **few-shot 품목 인식 PoC 실행 완료 — 가설 입증** — rectify(deskew+적응 quad+고정피치 그리드)로 행정렬 해결, 그룹핑 대응 라벨 25/36 trusted. 작성자 손글씨 retrieval **가능**: 다수라벨 베이스라인 13.3% → **손글씨 인코더(TrOCR) top-1 36%/top-3 60%**(자연이미지 DINOv2 20~25%). **인코더 도메인이 결정적**. 게이트(70/85%) 미달이나 한글 인코더+파인튜닝으로 가시권. 이후 사진 54장 추가 이관·통합(타임스탬프+검수 매칭, 74장 GT) + robustness 2패스: (a)**윈도우드 행선택**으로 하단 합계·메모 노이즈 오선택 교정, (b)**워프 코너검출 best-of-candidates**(축정렬 극점→회전불변 approxPolyDP/minrect/de-rotated 극점 결과채택; 회귀0) → trusted 49→**68**, clean crop **314**, 고유라벨 **142**; 전수검사 `crop_inspect.py`. 잔여 4(가림). few-shot은 30.4/44.1%로 **평탄** — 행정렬·워프는 해결, 남은 천장은 **이미지 품질·인코더**. (§8-B)
9. **측정 무결성·검출 완전성이 선결 리스크** — 위 84.8%·80.4%·4.4%p와 §8 게이트 임계는 채점이 grouping·정렬을 못 보는 set-recall이라 **잠정**이다. ① 채점을 DB 순서보존+연속합으로 재정의, ② 모델 선택을 GT-정렬 strip isolated 인식으로 재측정해야 확정(§8-A). 또한 end-to-end 운영 정확도는 인식 천장이 아니라 **검출 세로 완전성**(행 누락=복구불가 손실)에 캡되고, **작성자 단일성 가정도 38장으론 미검증**. (§7·§8)

## 1. 인식기 후보 벤치 (숫자 공급가)

**환경**: Apple M5 Max / 128GB, `mlx-vlm 0.6.3`(네이티브 `qwen3_vl`·`qwen2_5_vl`·`paddleocr_vl` 지원), 외부 API 0.
**채점**: GT 있는 36장의 **공급가(금액) 열 strip 1장**을 모델에 주고 "위→아래 금액 정수 리스트" → 천원 정규화 → DB 공급가(항목합)와 set 매칭. recall = 맞춘 DB 항목수 / 전체. ⚠️ 이 recall은 **검출 정렬 실패·grouping 미반영·인식 오류가 혼입된 end-to-end set-recall**이라 isolated 인식 정확도의 하한이다(§7 채점 재정의·§8-A).

| 모델 | 크기/정밀 | recall | 속도 | 비고 |
|---|---|---|---|---|
| **🥇 Qwen3-VL-8B** | 8B / 4bit | **173/204 = 84.8%** | 0.6s/장 | 신형. **잠정** 1순위 |
| 🥈 Qwen2.5-VL-7B | 7B / 4bit | 164/204 = 80.4% | 0.6s/장 | 기술리뷰 1순위. 근소 2위(차 4.4%p) |
| 🥉 PaddleOCR-VL | 0.9B / bf16 | 100/204 = 49.0% | 0.3s/장 | **`OCR:` 전용 프롬프트 필요**·빈출력/반복루프 불안정 |
| (참고) SP1 PP-OCRv5 | — | **~10%** | — | baseline |

- **Qwen3-VL-8B 잠정 채택**(Qwen2.5-VL 폴백). 단, 위 recall은 strip(검출 출력)에 매겨져 **검출 정렬 편차가 섞여 있고**, 4.4%p 격차는 n=36·신뢰구간 없음 → **인식 변별로 확정 불가**. 모델 선택은 GT-정렬 strip의 isolated 인식 정확도로 재측정해 확정한다(§8-A). PaddleOCR-VL은 chat-VLM이 아니라 레이아웃→OCR 파이프라인 전용이라 strip drop-in엔 부적합(한국어 지시 프롬프트 시 18%, `OCR:` 시 49%).
- 정렬된 strip은 자주 만점(inv_005 4/4, inv_011 9/9, inv_016 8/8, inv_017 9/9, inv_037 10/10) → **인식 천장은 84.8%보다 높다**(이 isolated 만점이 인식력의 더 정직한 신호). 미달분은 대부분 검출/grouping이며, 그만큼 end-to-end는 **검출 완전성에 캡**된다(§7).

## 2. 검출 고도화 — 템플릿 정합 (사용자 채택안)

**색 불변식 (핵심).** 양식은 38장 동일 **고정 인쇄 양식**이고, **격자선 = 연한 파랑(고정)**, **손글씨 = 검정 볼펜(펜색 가변)**. SP1 proto3가 회색조 모폴로지로 fragile했던 이유는 "격자·손글씨 색이 안 갈려서"였는데, 색차로 분리하면 해소된다.

- **파랑 마스크**: `(B - R) > 10 & V > 55`. 절대 HSV 채도보다 옅은 파랑(노출 밝은 inv_001)에 강함. 검정 손글씨·흰 종이 모두 배제.
- **form-quad(하이브리드)**: 마스크 `CLOSE(25)`로 구멍 메움 → 최대 contour. **극점(x±y) 코너**가 기본(원근 보존), 마주보는 변 길이비 < 0.72로 **깨지면 minrect 폴백**(배경 오염·옅은 코너로 한 코너가 안쪽으로 끌리는 경우 — inv_001). 순수 minrect는 강건하나 원근 손실로 전체 -3.4%p라 하이브리드가 정답.
- **원근보정**: `getPerspectiveTransform` → 캔버스 **900×2100**.
- **공급가 열 템플릿**(900×2100 기준): x ∈ **[612, 896]**, 데이터 y ∈ **[612, 1948]**. 38장이 같은 양식이라 워프만 일관되면 격자가 같은 좌표에 떨어짐(inv_010/020 열좌표 거의 동일로 입증).
- **EXIF 정위치 로드 필수**(`PIL.ImageOps.exif_transpose`) — cv2.imread는 EXIF 무시 → 양식 90° 누움.

코드: `apps/invoice-ocr/ml/report/sp2_spike/grid_v4.py`.

**검증**: 36장 중 약 30장 공급가 열 정렬. 미정렬은 배경 영수증 더미 위 촬영(동일 파란 양식 → 색 오염)·옅은 하단 사례.

## 3. 데이터 의미론 (반드시 반영)

**GT 출처.** references(인쇄 정본) OCR → 발행일 + 공급가합(천원·콤마 후보 중 최댓-유일hit) → DB `find_by_date_and_total_supply` 유일조회(36/38). DB는 **항목별 합산 공급가(원)**, 천원단위 정규화. (SP1 `match.py` 참조)

**약식 분해(grouping) 규칙.**
- 손글씨에서 **품목칸이 빈 행의 금액은 바로 위 품목에 합산**된다. DB는 합산값, 손글씨는 세부 내역.
- 실측 예 inv_001:
  - 손글씨 세부(위→아래): `100, 700, 270, 110, 120, 55, 60, 20, 40, 450, 44, 150` (12행)
  - 요소수호수 = 270+110+120+55 = **555**, 원터치 = 60+20+40 = **120**
  - DB 항목합(위→아래): `100, 700, 555, 120, 450, 44, 150` (7항목)
- **사람이 전산 입력할 때의 스타일 차이**라 결정론 규칙으로 흡수한다. 기술리뷰 §5의 "약식 분해 모호성"(inv_030: 4×100,000 vs 1×400,000)의 일반형.

**기타.**
- "unique 매칭"이 "정확 매칭"을 보장하지 않음 — grouping 미적용 set 채점은 일부 invoice를 과소채점(예: inv_001이 11/12 정확인데 2/7로 채점됨).
- **배경 클러터**: 일부 사진은 다른 영수증 더미 위에서 촬영(동일 파란 양식) → 색분리 오염 → quad 견고화로 대응.

## 4. 품목(품명) 인식 — 검증 결과 (2026-06-25 추가)

**결론: 품목 직접 OCR은 실패. 모델·크롭 문제가 아니라 본질적 가독성 한계.**

**벤치**(공급가와 동일 파이프라인, 품목 열 strip 36장). 품목 열 = 워프 900×2100 기준 x ∈ **[100, 392]**(수직 파랑선 검출로 확정: 할인|품목|수량|단가|공급가 경계 ≈ 105/386/510/630). GT = DB `invoice_items` 정식 품명(item_order 순).

| 대상 | 모델 | 결과 |
|---|---|---|
| 숫자 공급가 (§1) | Qwen3-VL-8B | **84.8%** |
| **품목 품명** | Qwen3-VL-8B | **loose(≥.5) 12.3% / tight(≥.7) 4.4%** |

- **왜 실패하나**(strip 직접 검수): 품목은 **개인 속기체**(`엔진오일→0IN`, `대우→대능`, `경유1차→겨기`, `도리까이→ㄷ리가먀`). 쓴 사람만 읽을 수 있고, **DB 정식명과 획 자체가 불일치** → OCR이 완벽해도 문자열 불일치. 숫자는 또박또박 써서 84.8%였던 것과 정반대.
- 일부 strip은 VLM이 반복 루프(`"소스"×75`)·환각(`비빔밥`,`가발`)에 빠짐 → 더 큰 모델로도 가독성 자체가 없어 안 풀림.
- **naive 잉크-빈칸 검출도 grouping 구조 회수 실패**(품목칸 채워진 행 == DB 항목수: 2/36). 행분할(hline)이 불안정(inv_001/019 0행, 다수 16행 과분할)·잉크 임계 거칢 → 정밀 작업 필요.
- 검수 시트: `apps/invoice-ocr/ml/review/item_review.html` (36장 strip · VLM 판독 · DB 정식명).

## 5. 작성자-특화 품목 인식 설계 (채택 방향)

직접 OCR이 실패한 이유는 *열린 한글 전사*로 접근해서다. 실제 구조는 **단일 작성자 · 닫힌 소규모 도메인 어휘**이므로 **"글씨 패턴 → 정식 품목명 분류/검색"** 문제로 재정의한다.

**마스터 어휘 = 분류 출력공간 (DB `invoice_items` 전체).**
- DB 400전표 / 1302인스턴스 / **고유 품명 386개**(DB 전체 쿼리로 실측). 129개 재현, **인스턴스의 80.3%(1045건)가 재현 품목**(엔진오일94·타이어92·공임70·세차54·구리스43…).
- → 인스턴스 가중으론 80%가 흔한 head라 표본이 빠르게 쌓인다. **단, 고유 품명의 66.6%(257개)가 싱글톤**이라 운영 첫인상을 좌우하는 *새 전표에 처음 보는 품목이 나올 확률*·*전표당 사람개입 행 수*는 인스턴스 20%보다 **높게 시작**한다 — 자동화율 기대치를 인스턴스 80%로 잡으면 어긋난다. 게이트도 인스턴스 가중뿐 아니라 **고유품명 커버리지·전표당 개입 행 수**로 함께 본다(§8-B).

**인식 아키텍처.**
- 출력공간 = DB 마스터 386(운영 중 증가).
- **exemplar 뱅크** = (손글씨 품목 crop ↔ 정식명) 누적.
- 인식 = query crop 임베딩 → 뱅크 최근접 랭킹 → top-k 후보 + 확률.
- **단계적 모델**: 지금은 **무학습 few-shot 최근접**(데이터 적음) → 표본 쌓이면 **파인튜닝** 승급(동일 데이터셋 재사용). 작성자 1인이라 글씨체가 안정적인 게 학습 전제(칸 이탈·약어도 패턴 고정이면 학습 대상).
- **⚠️ 이 '단일 작성자·글씨체 안정' 전제는 38장으로 미검증**이다 — 직원 교체·복수 작성자가 들어오면 exemplar 뱅크·파인튜닝이 동시에 무효화된다. PoC 0단계에서 38장이 실제 단일 작성자인지 1차 확인하고, 운영에선 **작성자 변경을 모델 무효화 트리거**로 둔다(§7).

**운영 루프.**

```
인식 → DB 마스터 리스트 최근접 후보+확률
   ├ 고신뢰 → 자동 채택
   └ 저신뢰 / 뱅크 미보유 → 사람이 품목등록 + 거래명세서(정답지) 등록
                              → 재학습 시 exemplar 뱅크 반영 (순환)
```

**데이터 계획 (핵심).**
- 손글씨 표본이 386 중 102개뿐인 건 한계가 아니라 **38장만 이관됐기 때문**.
- **PoC 테스트 후 손글씨 사진을 최대한 이관** → DB 400전표에 대응하는 손글씨 확보 시 마스터 386 대부분에 실표본 → 파인튜닝 연료.
- 38장 = PoC 검증셋, 확장 이관분 = 본 학습 데이터.

## 6. grouping·검산 방향 + 단가 제약 (갱신)

- **grouping = DB앵커 연속합 정합.** 연속 OCR 금액의 합 = DB 항목합이 되도록 매칭(약식 분해 흡수). ⚠️ 부분합 매칭이라 **같은 금액 반복·복수 해에서 틀린 그룹핑이 합검증을 통과**할 수 있고(false confidence), inv_030式(4×100,000 vs 1×400,000)은 합이 같아 **연속합으로 풀리지 않는다**.
- **의존 방향 끊기(중요).** 1차 파이프라인은 **품목 인식 없이** 동작해야 한다 — 품목 인식은 §8-B에서 아직 미검증이고, grouping이 품목에 의존하면 *순환*이다. 따라서 모호 케이스는 **사람 개입으로 처리**하고, 품목 인식 tie-breaker는 PoC 게이트 통과 후의 **선택적 보강**으로만 얹는다(품목→금액 오류 전파도 이 결합이 있을 때만 생긴다).
- **단가 변동 → 금액 역산 보류.** 단가는 시점별 변동 가능하므로 **금액은 손글씨 OCR(84.8%)에서만** 취한다. DB는 **합 검증 앵커**(항목합·grand_total)로만 쓰고, **DB 단가로 개별 금액을 유추하지 않는다**(우선 보류).

**A. 측정 무결성 (선결 — 이 위에서만 수치·게이트가 확정됨)**

1. **채점 재정의** — set-recall → **DB 순서 보존 + 연속합 정합**. 84.8%·80.4%·게이트 임계는 모두 이 채점 위에서만 확정된다(§7).

**B. few-shot 품목 인식 PoC (사용자 우선)**

4. **0단계: 작성자 단일성 확인** — 38장이 실제 단일 작성자인지 1차 검증(§5·§7).

## 8-B 결과 — few-shot PoC 1차 실행 (2026-06-25)

**파이프라인(검증 완료).** 4-코너 워프는 모서리만 맞춰 내부 격자선 잔여 기울기를 못 펴 per-item 행정렬이 무너졌다(inv_001=0선). 해결:
- **적응 robust quad** — 검출 contour가 폼 기대면적(<25%)을 못 채우면 파랑 임계를 단계 완화(밝은 노출 inv_001: area 0.10→0.37 복구). 작동 전표는 기본값 그대로(early-stop).
- **deskew** — Hough로 수평 파랑선 각도 측정→회전 평탄화(inv_001 14.5°).
- **고정 피치 행그리드 + 전표별 offset** — 양식 고정이라 행 피치 **P≈81px** 일정. deskew 후 φ(offset)만 맞추면 행이 고정좌표로 떨어짐(φ 전표간 일관 ≈58). → 행검출 자체가 불필요.
- **그룹핑 대응 라벨** — 빈 품목칸(연속행=위 품목 반복)은 stroke 낮음. 아는 DBn으로 **stroke 상위 DBn 행** 선택 + **연속블록이면 trusted**(no-grouping), 구멍 생기면 skip(흐림·grouping → 사람개입). → trusted **25/36**, clean crop **136**, 재현 라벨 **19**.

**few-shot leave-one-invoice-out (DINOv2 정사각 224, CLS / TrOCR 패치평균).** 인코더 도메인이 결정적:

| 인코더 | top-1 | top-3 |
|---|---|---|
| 다수라벨 베이스라인 | 13.3% | — |
| DINOv2-small (자연이미지) | 20.0% | 33.3% |
| DINOv2-base (자연이미지) | 25.3% | 36.0% |
| **TrOCR-base-handwritten (손글씨)** | **36.0%** | **60.0%** |

**판정·해석.**
- **가설 입증** — 단일 작성자 손글씨는 retrieval 가능(베이스라인 13.3% → 손글씨 인코더 36/60%, 명확한 실신호).
- **인코더 도메인이 핵심** — 자연이미지 인코더는 약하고, 손글씨 전용이 top-3를 60%로 끌어올림. crop 타이트화는 무영향(인코더가 병목).
- **상승 경로 명확** — TrOCR은 *영어* 손글씨 학습인데 *한글 속기*에 36/60%. 현재 게이트(top-1 70%/top-3 85%)는 미달이나, **한글 손글씨 인코더 + 작성자 파인튜닝 + 표본 확대**로 도달 가시권.
- **라벨 노이즈가 실교란** — 근사 정렬(12.4%) → 그룹핑 대응 정제(상위 DBn+연속블록) 후 동일 인코더 20.0%. 깨끗한 라벨이 +7.6%p.
- **운영 루프 검증** — 자동 trusted 25/36, 모호 11장은 사람개입 → §5 운영 루프와 일치.

**데이터 확대 — 사진 이관·통합 (2026-06-25).** 손글씨 사진 54장을 repo(`data/image/`, gitignore)로 추가 이관. 파일명 촬영일(`Resized_YYYYMMDD_HHMMSS`, KST)을 DB와 매칭:

- **타임스탬프 근접 매칭** — 파일명 시각(KST) ≈ DB `created_at`(UTC+9, 영수증 촬영 직후 입력). 같은 날 전표 여러 건이면 최근접 `created_at`을 **기본 제안**으로 미리 체크. 단 |Δ| 중앙 13분이라 분 단위 연속입력 전표는 타임스탬프 단독으론 못 가림(id 44/59 충돌 실증) → **인쇄 거래처·차량·공급가로 사람 검수**(검수 HTML, 라디오 확정). 기본값 **34/38 적중**, 사람 4건 교정.
- **고아 16장 제외** — 전부 8/1~8/16, DB 백업 최초 전표가 2025-08-18이라 매칭 근거 없음 → `_unmatched/` 별도 보관(과거 백업 확보 시 합류).
- **통합 정리** — 신규 38 + 기존 36(OneDrive, date+total_supply DB 유니크) = **74장** → `data/image_dataset/<date>_inv<id>.jpg`(날짜형 통일) + `manifest.json`. 라벨(GT)은 DB `invoice_items` 단일출처. 라벨셋 재빌드: trusted **49/74**, clean crop **214**, 재현 라벨 **31**(기존 25/136/19 대비 ~2배).
- **그러나 few-shot 하락(중요)** — 동일 TrOCR로 통합셋 **top-1 32.4%/top-3 42.6%** (기존 36/60% 대비 ↓). 양·재현라벨은 늘었으나 **새 사진 crop 품질이 뱅크를 오염**: 신규 사진은 파랑 격자가 흐리고 기울기·원근이 크며 2장 동시촬영이 섞여 rectify가 거칠다(inv049 15품목→0행, inv042 2장 워프). → **rectify robustness가 표본 확대의 선결조건**이다. §7의 검출 완전성(silent data loss)이 신규 사진에서 실제화한 것.

**robustness 패스 — 행선택 교정 + 전수검사 (2026-06-25).** crop 품질 전수검사(`crop_inspect.py`, 74장 원본↔워프+그리드↔추출crop + 실패모드 자동진단·필터)로 census를 냄: 지배적 실패는 **not_contig 23**(연속블록 깨짐). 진단 결과 — 양식 고정상 품목은 **헤더 직후부터 연속**으로 내려가는데, 기존 *전역 top-DBn stroke* 선택이 폼 **하단의 합계·전화메모·도장** 노이즈 행을 흐린 상단 실품목 대신 잡았다(inv032 [4–8]+노이즈11, inv169 하단 전화번호 픽업).

- **수정(윈도우드 선택)** — 헤더 이후 행에서 `DBn개 연속 + 전부 stroke≥TRUST_MIN` 윈도우의 **최상단**을 품목으로 선택. 하단 노이즈는 빈 행으로 분리돼 자연 배제. A/B 측정: top-DBn 49 → **윈도우드 64/74 trusted**(앵커드-런 변형은 13으로 과엄격, 기각). 회복 사례 inv032·inv169 오버레이로 라벨 안착 **육안 확인**(허수 아님).
- **결과** — clean crop 214→**299**, 고유라벨 109→**133**, 재현(≥2전표) 31→**38**. 잔여 skip 10장(no_window 9·count_short 1=inv049)은 흐린항목·grouping·15품목초과로 **정상 사람개입**.
- **그러나 few-shot은 평탄** — 통합셋 TrOCR top-1 30.4%/top-3 44.1%(top-DBn 32.4/42.6과 동률). 행선택은 고쳤으나 정확도 정체: 회복분이 더 흐린 전표라 뱅크 신호가 약하고, 라벨 공간이 2배(재현 38)로 커져 혼동↑. → **남은 레버는 행선택이 아니라 이미지 품질(deskew·2장분리)·인코더**다(few-shot 천장은 이미지/인코더가 캡).

**robustness 패스 — 워프 코너검출 교정 (best-of-candidates) (2026-06-25).** 사용자 지적: 워프+그리드가 일관되게 직사각으로 안 펴짐(inv011·038·066·087·146·018·045·102·135·153·216 등 양식 수직/수평선이 삐뚤). 근본원인을 증거로 규명 — **homography는 직선을 보존**하므로 코너만 맞으면 격자가 완벽한 직사각이 된다(워프 후 삐뚤면 코너가 틀린 것). GOOD/BAD 16케이스 측정: BAD는 워프 후 deskew각 큼 + **수평선 각도분산(hstd) 높음**(평행 아님=잔여 원근), GOOD은 둘 다 ≈0. 원본에 검출 quad 오버레이로 **육안 확정** — 기존 **축정렬 극점(min/max of x±y)** 코너검출이 회전·원근 사진에서 변 위 엉뚱한 점(특히 BL이 안쪽)을 집어 quad가 대각선으로 잘림(inv011 좌변이 폼을 가로지름). `edge_balance≥0.72`는 이 대각선 오류를 못 거른다(마주보는 변 길이비는 균형일 수 있어서 minrect 폴백도 안 걸림).

- **수정(best-of-candidates)** — 회전불변 후보 다수를 만들어 **결과로 고른다**: `extreme`(구버전 극점, near-axis 보존)·`approxPolyDP`(convex hull 4꼭짓점, dilate/tight blob)·`minAreaRect`(tight)·**`de-rotated 극점`**(minAreaRect 각도로 펴서 극점법 적용 후 역회전 — 회전+원근 동시 케이스의 사다리꼴 코너를 정확히). 각 후보를 워프+deskew해 **직사각성 점수 = DATA_Y full-width 수평선 수 − 2.5·hstd**(선완성도+직진성, 가중 2.5로 직진 우대)로 최댓값 채택. tight blob(`dilate 9×9×2` 제거)이 코너 부풀림(~18px)을 없애 최다 채택. extreme를 첫 후보로 둬 동점 시 구버전 byte-동일 보존.
- **결과** — 전체 74장 **다수 개선·회귀 0**(extreme이 항상 후보라 구조적 무회귀). inv011 hstd 4.78→**0.26**, inv087 1.79→**0.50**, inv066 3.06→**0.27**, **inv050 2.02→0.41**(de-rotated). 워프 육안 확인: 전단되던 폼이 정립 직사각. 다운스트림 — trusted 49→**68**, clean crop **314**, 고유라벨 **142**(few-shot 어휘↑). **잔여 미해결 4(inv045·018·074·135)**: 영수증이 동종 파랑격자 종이더미 위에 겹쳐 blob이 더미로 번지는 **가림(occlusion)** — 코너검출 범위 밖, 근본해는 재촬영 가이드(plain 대비 배경). homography 자체 한계라 전부 개선·동등이되 완전치 않음.

**robustness 패스 — 2장 동시촬영(연속 영수증) 정규화 검토 (2026-06-26).** 한 사진에 영수증 두 매가 나란히 찍힌 1건(inv042, DB 13품목)을 검토. **정규화 가능 — 스키마 변경 0**: 거래명세서 1건 = invoice 1건 = **순서 보존 단일 품목리스트**이고(DB가 이미 그렇게 모델·photomatch도 사진↔invoice 1:1), 물리적 장수는 의미 없는 표현상 오버플로우다. 규칙 — **우측 시트는 좌측의 연속**이므로 시트를 읽기순서(좌→우, 각 시트 내 위→아래)로 이어붙이면 DB 순서에 정렬된다(§4 빈칸=위행합산 grouping의 **가로 아날로그**; 연속 시트의 합계칸은 공란 — inv042 우측 합계 공란으로 일치).

- **현재 능동 오염(실측)** — `form_quad_robust`가 두 영수증을 한 quad로 묶음(6후보 전부 asp 0.88~0.91 vs 단일 0.43, rect_score 전부 음수). 병합워프를 900×2100에 욱여넣어 격자 어긋남에도 `select_items`가 **trusted=True**로 판정 → 13개 **오정렬 crop**이 킹핀·하부·라이닝… 라벨로 dataset_v2에 주입돼 **뱅크를 능동 오염**(§7 silent data loss의 실교란). 전수검사 "2장워프의심 3" 중 진짜 연속은 **inv042 단독**(inv054=단일 거래명세표 붉은배경, inv153=단일 영수증 근접촬영+옆 노트 — 둘 다 오탐, asp>0.6 정밀도 1/3).
- **수동 분할 수확 검증** — 좌/우 대략 박스를 잘라 각각 `form_quad_robust`하면 단일 영수증으로 분리됨(asp 0.90→0.48/0.51, 워프 육안 양호). 그러나 절반워프는 파랑 격자선이 옅어 hline **1~2개**뿐 → `fit_phase` 위상잠금 실패(상단 연속채움 자동집계 12+10=22≠13). **템플릿 고정 위상(φ=Y0%P)** 폴백 시 좌측은 대체로 안착하나 **우측은 거침** — 박스+시트수만으론 **셀-완벽 crop 미달**(흐린격자 한계, §199의 "남은 레버=이미지 품질" 재확인).
- **조치** — `dataset_build.py`가 `twoup_split.json` 등재 cname을 정상경로에서 **제외**(병합워프 13 라벨노이즈 차단; 이게 핵심 수정). 수확은 `twoup.py`(박스 분할 수확·검수 HTML)로 분리. **권고: 뱅크에서 inv042 제외**(품질>1건 수량; 근사 crop은 §185식 라벨노이즈 위험). `twoup.py`는 박스/시트수 튜닝 또는 **재촬영(1장=1시트, 근본해)** 시 13crop 복구용 도구로 보존.

**다음.** (1) 이미지 품질 robustness — 워프 코너검출·**2장 분리 검토 완료**(위), 남은 건 **가림 4케이스+2장 셀정렬의 공통 선결조건 = hline/위상 robustness**(흐린격자)·재촬영 가이드. (2) 한글 손글씨 인코더 / 작성자 파인튜닝(이제 **314 clean crop·142 라벨** 확보 — 파인튜닝 연료 충분). (3) 그룹핑 전표 라벨 amount 연속합 앵커 확장(§6). (4) 게이트 비용 기준(§8-7).
