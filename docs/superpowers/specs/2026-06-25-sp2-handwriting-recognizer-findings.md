# SP2 손글씨 인식기 — 스파이크 결과 및 방향 (2026-06-25)

> SP1 게이트(인식이 단일 병목)를 받아, 손글씨 특화 인식기를 리서치·벤치하고 셀
> 검출을 고도화한 스파이크 기록. **다음 세션 SP2 본구현(품목 열 + grouping)** 의 입력.
> spec/plan SJMJ-Web 미러링 불필요(로컬 전용).

---

## 0. TL;DR

1. **로컬 VLM이 SP1 병목을 깼다** — M5 Max에서 MLX로 **Qwen3-VL-8B = 84.8%**(공급가 열 36장, DB 대비 recall) vs SP1 stock PP-OCRv5 **~10%**. 외부 API 0, 0.6초/장.
2. **검출 = 연파랑 격자 색분리 + 템플릿 원근보정**. SP1이 fragile로 접은 괘선 추출을, "격자=연파랑(고정)·손글씨=검정(가변)"이라는 **색 불변식**으로 돌파. 공급가 열 strip을 안정 추출.
3. **GT 의미론 확정** — references→DB 정답은 **항목별 합산 공급가**. 손글씨↔DB 차이의 정체는 **약식 분해(grouping)**: 품목칸이 빈 행의 금액은 **바로 위 품목에 합산**. (인식 오류 아님)
4. **남은 갭은 인식이 아님** — ① 검출 세로 완전성(옅은 하단), ② grouping 입도(품목 열 필요), ③ 가끔 단자리 오독. 모두 처리 가능.
5. **다음 세션(대기) = 품목 열 + grouping 규칙 + DB 앵커 검산.**

## 1. 인식기 후보 벤치

**환경**: Apple M5 Max / 128GB, `mlx-vlm 0.6.3`(네이티브 `qwen3_vl`·`qwen2_5_vl`·`paddleocr_vl` 지원), 외부 API 0.
**채점**: GT 있는 36장의 **공급가(금액) 열 strip 1장**을 모델에 주고 "위→아래 금액 정수 리스트" → 천원 정규화 → DB 공급가(항목합)와 set 매칭. recall = 맞춘 DB 항목수 / 전체.

| 모델 | 크기/정밀 | recall | 속도 | 비고 |
|---|---|---|---|---|
| **🥇 Qwen3-VL-8B** | 8B / 4bit | **173/204 = 84.8%** | 0.6s/장 | 신형. 1순위 |
| 🥈 Qwen2.5-VL-7B | 7B / 4bit | 164/204 = 80.4% | 0.6s/장 | 기술리뷰 1순위. 근소 2위 |
| 🥉 PaddleOCR-VL | 0.9B / bf16 | 100/204 = 49.0% | 0.3s/장 | **`OCR:` 전용 프롬프트 필요**·빈출력/반복루프 불안정 |
| (참고) SP1 PP-OCRv5 | — | **~10%** | — | baseline |

- **Qwen3-VL-8B 채택.** Qwen2.5-VL은 폴백. PaddleOCR-VL은 chat-VLM이 아니라 레이아웃→OCR 파이프라인 전용이라 strip drop-in엔 부적합(한국어 지시 프롬프트 시 18%, `OCR:` 시 49%).
- 정렬된 strip은 자주 만점(inv_005 4/4, inv_011 9/9, inv_016 8/8, inv_017 9/9, inv_037 10/10) → **인식 천장은 더 높다**. 미달분은 대부분 검출/grouping.

## 2. 검출 고도화 — 템플릿 정합 (사용자 채택안)

**색 불변식 (핵심).** 양식은 38장 동일 **고정 인쇄 양식**이고, **격자선 = 연한 파랑(고정)**, **손글씨 = 검정 볼펜(펜색 가변)**. SP1 proto3가 회색조 모폴로지로 fragile했던 이유는 "격자·손글씨 색이 안 갈려서"였는데, 색차로 분리하면 해소된다.

- **파랑 마스크**: `(B - R) > 10 & V > 55`. 절대 HSV 채도보다 옅은 파랑(노출 밝은 inv_001)에 강함. 검정 손글씨·흰 종이 모두 배제.
- **form-quad(하이브리드)**: 마스크 `CLOSE(25)`로 구멍 메움 → 최대 contour. **극점(x±y) 코너**가 기본(원근 보존), 마주보는 변 길이비 < 0.72로 **깨지면 minrect 폴백**(배경 오염·옅은 코너로 한 코너가 안쪽으로 끌리는 경우 — inv_001). 순수 minrect는 강건하나 원근 손실로 전체 -3.4%p라 하이브리드가 정답.
- **원근보정**: `getPerspectiveTransform` → 캔버스 **900×2100**.
- **공급가 열 템플릿**(900×2100 기준): x ∈ **[612, 896]**, 데이터 y ∈ **[612, 1948]**. 38장이 같은 양식이라 워프만 일관되면 격자가 같은 좌표에 떨어짐(inv_010/020 열좌표 거의 동일로 입증).
- **EXIF 정위치 로드 필수**(`PIL.ImageOps.exif_transpose`) — cv2.imread는 EXIF 무시 → 양식 90° 누움.

코드: `apps/invoice-ocr/ml/report/sp2_spike/grid_v4.py`.

**검증**: 36장 중 약 30장 공급가 열 정렬. 미정렬은 배경 영수증 더미 위 촬영(동일 파란 양식 → 색 오염)·옅은 하단 사례.

## 3. 데이터 의미론 (반드시 반영)

**GT 출처.** references(인쇄 정본) OCR → 발행일 + 공급가합(천원·콤마 후보 중 최댓-유일hit) → DB `find_by_date_and_total_supply` 유일조회(36/38). DB는 **항목별 합산 공급가(원)**, 천원단위 정규화. (SP1 `match.py` 참조)

**약식 분해(grouping) 규칙 — 다음 세션 구현 대상.**
- 손글씨에서 **품목칸이 빈 행의 금액은 바로 위 품목에 합산**된다. DB는 합산값, 손글씨는 세부 내역.
- 실측 예 inv_001:
  - 손글씨 세부(위→아래): `100, 700, 270, 110, 120, 55, 60, 20, 40, 450, 44, 150` (12행)
  - 요소수호수 = 270+110+120+55 = **555**, 원터치 = 60+20+40 = **120**
  - DB 항목합(위→아래): `100, 700, 555, 120, 450, 44, 150` (7항목)
- **사람이 전산 입력할 때의 스타일 차이**라 결정론 규칙으로 흡수한다. 기술리뷰 §5의 "약식 분해 모호성"(inv_030: 4×100,000 vs 1×400,000)의 일반형.

**기타.**
- "unique 매칭"이 "정확 매칭"을 보장하지 않음 — grouping 미적용 set 채점은 일부 invoice를 과소채점(예: inv_001이 11/12 정확인데 2/7로 채점됨).
- **배경 클러터**: 일부 사진은 다른 영수증 더미 위에서 촬영(동일 파란 양식) → 색분리 오염 → quad 견고화로 대응.

## 4. 남은 갭 / 리스크

| 갭 | 내용 | 대응 |
|---|---|---|
| 검출 세로 완전성 | 옅은 하단에서 form-quad가 양식을 짧게 잘라 하단 행 누락(inv_001: 12행 중 7행만 strip에 담김) | form-quad 하단 보강(하단 격자 약신호 보정) / DATA_Y 동적화 |
| grouping 입도 | 현재 공급가 열만 읽음 → DB 항목합과 정합하려면 품목 열도 읽어 빈칸 grouping 필요 | §5-1 (다음 세션) |
| 단자리 오독 | 450, 365→364 등 | 고해상 crop / DB 앵커 검산 복원 |
| 채점 방법 | set 매칭은 grouping을 못 봄. 연속합 매칭(crude)도 DB 순서 미보존이라 +0.5%에 그침 | 채점을 **DB 순서 보존 + 연속합 정합**으로 재정의 |

## 5. 다음 세션 스텝 (대기) — SP2 본구현

> 사용자 결정(2026-06-25): "3번(이 정리) 하고, 1번(품목 열+grouping)은 다음 세션 새 스텝으로 대기."

1. **품목 열 검출·인식 추가** — 공급가 열 좌측의 품목 열도 crop·인식 → **빈 품목 행 식별**.
2. **grouping 정규화** — 빈 품목 금액을 바로 위 항목에 합산 → 항목합 산출.
3. **DB 앵커 검산** — DB 항목합 + grand_total을 앵커로 grouping 검증·복원(상호보상 오류는 신뢰도 게이트로만).
4. **Qwen3-VL을 `RecognizerAdapter` 뒤에 통합** — `mlx-vlm`을 `[ml]`처럼 **별도 optional extra**로(코어 paddle-free 유지). 기존 SP1 파이프라인에 plug-in.
5. **검출 세로 완전성 보강** — 옅은 하단 form-quad 확장.
6. **채점 재정의** — grouping-aware(DB 순서 보존 + 연속합).
