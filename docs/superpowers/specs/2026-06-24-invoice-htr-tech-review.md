# 수기 거래명세서 인식 — 기술검토 보고서 (VLM 실패 후 아키텍처 재설계)

- 작성일: 2026-06-24
- 입력: 멀티앵글 리서치 워크플로(6차원 조사 + 아키텍처 설계 + 3렌즈 adversarial 검증, 에이전트 10개)
- 배경: 기존 PoC 설계(2026-06-24 "외부 VLM API 벤치마크")의 전제가 사용자 피드백으로 무효화됨
  - **VLM 단순 추출은 이미 Claude/GPT로 시도 → 형편없었음**
  - **외부 상용 API 키 사용 계획 없음** (비용)
  - 가용 자원: macmini의 hermes(codex oauth, 비전) + on-prem 로컬 모델
  - 사용자 두 아이디어: ① codex가 좌표검출 + 학습모델이 내용추론 하이브리드, ② 전용 이미지모델 학습

## 1. 결론 요약 (TL;DR)

1. **VLM 통짜 추출이 형편없던 건 우리 잘못이 아니라 구조적 한계** — 학계가 같은 현상을 보고. 통짜 MLLM은 손글씨 재무문서에서 F1 54.55%, 문서단위 0%. 재시도는 무의미.
2. **사용자 아이디어는 방향이 옳다. 단 "검출을 codex에 맡긴다"는 부분만 교체** — VLM의 좌표 grounding은 매우 약함(5~21%). 검출은 전용 detector(YOLO, mAP 0.89)에 맡기고, codex/VLM은 내용·검증으로 한정.
3. **추천 아키텍처 = B(모듈분리 하이브리드)**: `검출(YOLO) → 숫자 전용 CTC + 품목 open-vocab 인식 → 약식정규화·산술검산(symbolic)`. 완전 on-prem, 외부API 0, 비용 0. 사용자 아이디어의 정제판.
4. **단, 검증 패널이 기대치에 강한 경고** — "90% 정확도" 같은 수치는 현 근거로 과대평가. PoC를 "목표 달성 검증"이 아니라 **"라벨 신뢰도 → raw 인식 → 검산 게인 → 합성 갭"을 순차 측정해 기대치를 데이터로 교정하는 절차**로 운영해야 함.

## 2. 사용자 두 아이디어 평가

### 아이디어 ① "codex가 1차 좌표검출 + 학습모델이 내용추론" → **70% 채택 (검출 주역만 교체)**

- ✅ **검출/인식 분리**라는 큰 그림은 학계·산업 표준과 정확히 일치(KORIE, ICCV2025W 모두 3단 분리).
- ❌ **검출을 codex/VLM에 맡기는 건 위험**. 근거:
  - 문서 좌표 grounding: GPT-4o **5.27%**, Qwen2.5-VL-7B 21.5%, Gemini 0~28% (DOGR-Bench, ICCV2025)
  - 정밀 IoU0.95에서 VLM은 한 자릿수로 붕괴, 전용 detector DocLayout-YOLO는 **91.2** (LocateAnything, NVIDIA)
  - 우리는 인접 셀이 가까워 좌표가 정밀해야 함 → VLM 검출은 셀이 섞임
- ✔️ **수정안**: 검출 = DocLayout-YOLO/YOLOv11 fine-tune (KORIE 한국영수증 mAP@0.5 **0.888**), codex는 검증·예외처리·합성라벨로.

### 아이디어 ② "전용 이미지모델 트레이닝" → **방향 채택, 단 단독 end-to-end는 부적합**

- ❌ **순수 end-to-end 학습(Donut/TrOCR 페이지 통짜)은 소량데이터에서 붕괴**(stock TrOCR 손글씨 CER **48%** 사례). 생성형 디코더라 숫자 환각 위험 — 우리가 이미 겪은 실패 패턴 재연.
- ✔️ **올바른 "전용모델"은 모듈분리형**: 숫자=작은 charset CTC(환각 원천 차단), 품목명=open-vocab 인식, 약식→정규화=결정론 규칙. 이게 아키텍처 B.

## 3. 추천 아키텍처 B — 모듈분리 하이브리드 (사용자 아이디어 정제판)

```
손글씨 사진
  │ de-skew/원근보정 (OpenCV)
  ▼
[검출] DocLayout-YOLO/YOLOv11 fine-tune → 행·셀 bbox
       + 인쇄 격자선은 OpenCV Hough/투영으로 열(column) 룰 고정   ← "룰=열, 학습=행"
  │ 셀 crop (1920×2560 원해상도 보존)
  ▼
[인식]  숫자 셀 → PaddleOCR CTC (charset 0-9·천단위·〃 제한) ── 환각 원천 차단
        품목 셀 → korean_PP-OCRv5_rec (14MB, open-vocab 문자열 생성)
  │ raw 텍스트
  ▼
[symbolic IE] 약식 정규화(천원곱·'25×4' 압축 파싱·ditto 전파·빈칸=0)
              + 산술검산 게이트(수량×단가=금액, 세액=공급가×10%, 합계 검증·역산 복원)
  │
  ▼
[정규화] 품목명 사전(item_suggestions+운영DB) fuzzy/임베딩 매칭 + 신규품목 HITL 큐
  ▼
구조화 JSON {rows[], grand_total}   /  검산 실패·저신뢰 셀 → 사람 검수 큐
```

**왜 B인가 (우리 5개 제약에 정조준)**

| 제약 | B의 충족 방식 |
|------|---------------|
| 외부 API 불가 | 전 구성요소 on-prem, 비용 0 |
| VLM 통짜 실패 | 애초에 전용 인식기 — 통짜 안 함 |
| 소량 데이터(38쌍/211행) | 라벨에 셀 bbox 있음 → 211행을 셀 crop 수백~천개로 증폭. YOLO·CTC는 사전학습 위 fine-tune이라 소량 수렴 |
| 단일 필자·고정 격자 | writer-specific fine-tune 유리(16라인 25%/256라인 50% CER 개선). 격자선은 룰로 데이터 0에 즉시 활용 |
| 산술검산 무기 | symbolic Validator로 1급 배치. 일부 셀만 맞아도 역산 복원 |

**보조 노선**: A(on-prem Qwen2.5-VL 셀 crop read)와 D(codex 조율)는 **동일 채점셋에 꽂는 "교체 가능 인식 백엔드"**로 비교 측정. C(TrOCR/DTrOCR)는 흘림체 품목명 보조 인식기로만.

## 4. 우리 데이터의 결정적 유리 조건 (자체 분석 + 리서치 결합)

| 사실 | 함의 |
|------|------|
| 단가·금액 **100% 천원 배수**(211/211) | 손글씨 숫자가 5~6자리가 아니라 **2~3자리**. CTC 인식 난이도 급감 |
| **산술관계 100% 성립**(211/211) | 일부 셀만 읽어도 역산 복원. raw 인식보다 최종 필드정확도가 구조적으로 높음 |
| **단일 필자** | writer-specific 과적합이 오히려 목표와 일치(운영도 같은 1인) |
| 라벨 **211행 bbox 보유** | 셀 crop 자동 추출 → CTC 학습셋 즉시 확보 |
| **정본(references) + 사전 + 운영DB 보유** | 약식↔정규화 페어 **대량 합성** 가능 (Manchu 선례: 60k 합성 → 손글씨 CER 2.19%) |
| 수량 **91%가 1** | 대부분 단가=금액, 인식 부담 낮음 |

## 5. ⚠️ 검증 패널이 잡은 냉정한 경고 (과대평가 방지 — 반드시 반영)

adversarial 3렌즈 검증에서 설계의 낙관을 다음과 같이 깎았다. **이게 이 보고서에서 가장 중요한 부분.**

1. **라벨 신뢰도가 미검증 (최상위 리스크)** — 라벨은 `label_parser` 생성물이라 오류 가능(기존 spec도 명시). 38쌍 라벨을 references 정본과 **셀 단위 교차검증해 라벨 오류율을 먼저 측정**해야 함. 이 수치가 모든 KPI의 진짜 천장.
2. **라벨=정규화값이라 채점 불일치** — inv_001은 단가칸이 **사진에 아예 없음**(역산된 라벨값). raw read로 단가 채점은 불가능. → 채점 스키마를 **"사진에 실재하는 필드만 raw 채점, 나머지는 역산 일치로만"** 재정의.
3. **"90% 정확도" 기대는 overstated~uncertain** — 실제 손글씨 숫자열 벤치(ORAND) 92~94%, 우리 약식은 더 어려움 → string 완전일치 **85~93%가 현실적**. 품목명은 사전 밖 62%가 전체를 끌어내릴 위험.
4. **합성 domain gap** — 설계가 낙관 인용한 Extend.ai 사례가 실은 반례: 합성/공개 80% → 실데이터 **58% 하락**, 메우는 데 실라벨 6,000장 필요. 합성은 "만능"이 아니라 **"검증할 가설"** — held-out 실손글씨 테스트로 갭을 측정해야.
5. **산술검산 거짓양성** — 상호보상 오류(수량↑·단가↓)는 곱이 같아 검산 통과하지만 둘 다 틀림. **검산은 "정답 보증"이 아니라 "신뢰도 게이트"로만**. pseudo-label 자동채택은 다중 셀 독립 일치 시에만.
6. **2셀 동시오류/행누락 시 역산 미결정** — 약식 압축·빈칸으로 "단일 셀만 오류" 가정이 깨지기 쉬움. 검산 레버는 조건부.

**실현가능성 검증의 발견 (자원 재평가)**
- ✅ **macmini = M5 Max / 128GB 실측** → on-prem VLM(Qwen2.5-VL-7B)을 충분히 구동·학습 가능. "메모리 병목" 서사 폐기. A노선 위상을 보조 → 본선급 재평가.
- ⚠️ **codex(D)는 5시간/주간 메시지 한도** → 배치·반복실험·운영 Validator 부적합. **합성 라벨 부트스트랩·소수 셀 자문**으로만 한정.
- ⚠️ **진짜 병목은 흘림체 한국어 품목명** → 검출·검산보다 먼저 stock 모델로 품목명 baseline CER을 측정해 "가장 불확실한 가정"부터 깨라.

## 5b. 0단계 데이터 감사 결과 — 운영DB 교차검증으로 리스크 3개 해소 (2026-06-24)

운영 DB 백업(`database/db-2026-06-24-backup.sql`, **invoices 400건 / invoice_items 1,302행**, 2025-08-18~2026-06-24)을 정답지로 §5의 경고를 **실측 검증**했다. 매칭 키 = `grand_total`(38개 라벨 중 37개 매칭).

**label_parser 라벨 정확도** (라벨 34건/197행 vs 운영DB 정답):

| 필드 | 일치율 | 비고 |
|------|--------|------|
| 공급가액 | **100.0%** | 라벨 `amount` = DB `supply` 확정 |
| 수량 | 99.5% / 단가 99.5% | 각 1건만(분해 차이) |
| 품목명 | 94.4% | 불일치 대부분 표기변형(가스켓/가스겟, 밸브/발브) |

**§5 리스크 재평가:**

| §5 경고 | 감사 결과 |
|---------|-----------|
| 라벨 신뢰도 미검증 (최상위) | ✅ **해소** — 숫자 99.5~100% 정확 |
| 사전 밖 62% (open-vocab 병목) | ✅ **해소** — 운영DB **386종**으로 라벨 품목 **92% 커버** |
| 합성 domain gap / 실라벨 부족 | ✅ **해소** — **400건 실정답** + 사진만 받으면 `grand_total`+`issue_date`로 자동 페어화 |

**새로 확정된 규칙 (채점·학습 설계에 반영, PoC spec v2 §3·§6 참조):**
- 라벨 `amount` = **공급가액**, DB `total`은 부가세 포함 → 정답·채점은 **공급가 기준** 통일
- 품목명은 **DB조차 표기 흔들림** → **정규화(canonical) 레이어 필수**, 채점도 정규화 후 일치
- **약식 분해 모호성 실재**(inv_030: 라벨 `4×100,000` vs DB `1×400,000`, 공급가 동일) → 채점 무게를 (수량,단가)보다 **(공급가,금액)**에
- **사진↔DB 매칭 키 = `grand_total` + `issue_date`** → 빈 날짜 사진 추가 시 자동 정답화

**남은 리스크(§5에서 여전히 유효)**: 약식 분해 모호성, 흘림체 품목명이 진짜 병목, 산술검산 거짓양성(상호보상 오류) → "정답 보증" 아닌 "신뢰도 게이트"로만.

## 6. 권장 PoC 경로 (검증 반영 수정판)

> 핵심 전환: PoC의 질문을 "얼마나 정확히 읽나"에서 **"사진에 실재하는 정보로 검산까지 통과시켜, 입력보조 시 사람 수정부담을 얼마나 줄이나"**로 재정의.

- **0단계 — 데이터 진실 감사 (추가 데이터 0, 구축 전 필수 게이트)**
  - 라벨 vs references 셀 단위 교차검증 → **라벨 오류율 %** 확보 (verify: KPI 분모 확정)
  - 각 필드가 "사진에 실재 vs 역산값"인지 사람 태깅 → 채점 스키마 재정의
  - 약식 예외율 통계('25×4'·'〃'·빈칸·천원생략 비일관) → symbolic 규칙 카탈로그
  - 셀 crop 자동 추출 + Hough 열 경계 추출 성공률
  - codex 외부전송 거버넌스 확인 (D 가용성)

- **1단계 — B-축소 골격 + 검산 게이트 (단일 경로)**
  - 라벨 bbox를 **oracle로 사용**(검출 노이즈와 인식 품질 분리 측정)
  - 숫자 셀 PaddleOCR CTC raw read → symbolic 정규화 → 산술검산
  - **raw read**와 **read+검산** 두 축으로 측정 (verify: 숫자 string 정확도, 검산 후 행 정합)

- **2단계 — 인식 백엔드 비교** (동일 채점셋·동일 검산 게이트)
  - (A) on-prem Qwen2.5-VL-7B 셀 crop read / (CTC) / (D) codex 조율(거버넌스 허용시) / (C) TrOCR 품목명 보조
  - verify: 백엔드별 셀·행·명세서 완전일치율 표 → "숫자=?, 품목명=?" 최적 비대칭 확정

- **3단계 — 데이터 증폭** (실현가능성 입증 후)
  - 약식↔정규화 합성 페어(정본+사전 기반) + AI Hub 한글 손글씨 사전학습 + 산술검산 weak supervision
  - **held-out 실손글씨로 domain gap 측정** (합성 맹신 금지)

- **4단계 — 운영 판정**
  - 자동확정 비율·HITL 부하로 운영모드 결정 (입력보조 통합 / 무인 / 보류·데이터수집)

## 7. 핵심 기술 스택 (전부 on-prem·무료·오픈)

| 단계 | 1순위 | 비고 |
|------|-------|------|
| 검출 | DocLayout-YOLO(DocSynth-300K 사전학습) / YOLOv11 | HF 가중치, macmini 구동, KORIE mAP 0.888 |
| 숫자 인식 | PaddleOCR CTC (SVTR/PP-OCRv4_rec, charset 제한) | 10.5MB, Apache-2.0, 환각 차단 |
| 품목명 인식 | korean_PP-OCRv5_mobile_rec | 14MB, 한국어 전용, open-vocab |
| 보조 인식기 | TrOCR-large / DTrOCR + LoRA | 흘림체 품목명만, GPU 권장 |
| on-prem VLM(백엔드 비교) | Qwen2.5-VL-7B (MLX/Ollama) | M5 Max 128GB로 구동·QLoRA 가능 |
| 사전학습 데이터 | AI Hub '대용량 손글씨 OCR'(106,519장)·'다양한 형태 한글 OCR'(117만) | 무료(내국인) |
| 정규화 | 사전 + zero-shot entity linking(NEN) | 신규품목 재학습 없이 흡수 |
| 검산 | 자체 symbolic Validator | 산술 항등식 |

**선례(존재 증명)**: 한국에서 **비표준 거래명세서 손글씨 품목·금액·수량 인식**을 한글 손글씨 데이터셋으로 전용 OCR을 학습해 해결한 사례가 이미 있음(Superb AI). 우리 과제 형태가 풀린다는 강한 근거.
