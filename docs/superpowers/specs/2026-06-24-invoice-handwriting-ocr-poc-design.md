# 수기 거래명세서 → DB 자동입력 PoC 설계 (v2 — 전면 개정)

- 작성일: 2026-06-24 (v2 개정: 기술검토 + 0단계 데이터 감사 반영)
- 상태: 설계 개정 — 구현 plan 작성 전
- 범위: **PoC(실현가능성·정확도 검증)** — 운영 통합은 PoC 결과를 본 뒤 판단
- 기술 근거 문서: [`2026-06-24-invoice-htr-tech-review.md`](./2026-06-24-invoice-htr-tech-review.md) (리서치·아키텍처·검증 종합)

> **v1 → v2 변경 요지**: v1은 "외부 VLM API로 통짜 추출해 정확도를 잰다"였다. 그러나 ① 외부 상용 API 사용 안 함 ② VLM 통짜 추출은 이미 실패(학계도 동일 보고) ③ 운영 DB 백업이 정답지로 확보됨 — 이 세 가지로 방향을 **"on-prem 모듈분리 하이브리드 + 운영DB 정답지 기반"**으로 전면 전환한다.

## 1. 목적

성진자동차(SJMJ)의 수기 거래명세서(인쇄 영수증 양식 + 손글씨) 사진에서
품목·수량·단가·공급가·총액을 구조화 데이터로 추출하는 자동화의 **실현가능성을 검증**한다.

PoC가 답할 질문 (v1에서 재정의):

> ~~"VLM으로 얼마나 정확히 뽑나"~~ →
> **"사진에 실재하는 정보를 on-prem 전용 인식기로 읽고 산술검산까지 통과시켜,
> 입력보조 시 사람 수정 부담을 얼마나 줄일 수 있는가?"**

성공 기준선은 미리 박지 않는다. 단계별 메트릭(라벨 신뢰도 → raw 인식 → 검산 게인)을 측정해 운영 적용 가능성을 해석한다.

## 2. 데이터 자산

| 자산 | 내용 | 역할 |
|------|------|------|
| `images/` (OneDrive) | 수기 원본 사진 38장 (1920×2560) | 입력 |
| `labels/` (OneDrive) | label_parser JSON 38개 (bbox+텍스트, 211행) | 샘플 정답·검증 |
| `references/` (OneDrive) | 디지털 정본 38장 | 참고 |
| **`database/db-2026-06-24-backup.sql`** | **운영DB 백업: invoices 400건 / invoice_items 1,302행 (2025-08-18~2026-06-24)** | **정답지(ground truth) 본체** |

**핵심 데이터 전략**: 운영 DB가 학습 정답지다. 사진은 거래처/날짜가 없고 차량번호·품목·총액만 있으므로,
**사진↔DB 매칭 키 = `grand_total`(합계금액) + `issue_date`**. 빈 날짜 사진을 추가로 받으면 날짜+금액으로 DB invoice를 찾아 **(사진, 정답) 페어를 자동 생성** → 수동 라벨링 없이 학습 데이터가 축적된다. (38개 라벨 중 37개가 `grand_total`로 DB와 매칭 확인됨)

## 3. 0단계 데이터 감사 — 확정 사실 (운영DB 교차검증 완료)

label_parser 라벨 34건/197행을 운영DB와 교차검증한 결과:

| 항목 | 결과 | 함의 |
|------|------|------|
| **라벨 숫자 정확도** | 공급가 100% · 수량 99.5% · 단가 99.5% | 라벨은 믿을 만함 (최상위 리스크 해소) |
| **라벨 품목명** | 94.4% (불일치 대부분 표기변형: 가스켓/가스겟, 밸브/발브) | 정규화 후 비교 필요 |
| **필드 의미** | 라벨 `amount_text` = **공급가액**(= DB `supply`), DB `total`은 부가세 포함 | 채점·학습 정답은 **공급가 기준** 통일 |
| **숫자 약식** | 단가·금액 100% 천원 배수, 수량×단가=공급가 100% 성립 | raw는 2~3자리, 산술검산 강력 |
| **품목 사전** | 운영DB 386종이 라벨 품목의 **92% 커버** | open-vocab 부담 급감 (사전 밖 62%→8%) |
| **약식 분해 모호성** | inv_030: 라벨 `4×100,000` vs DB `1×400,000` (공급가 동일) | 채점 무게를 (수량,단가)보다 **(공급가,금액)**에 |

## 4. 아키텍처 — B: on-prem 모듈분리 하이브리드 (사용자 아이디어 정제판)

```
손글씨 사진
  │ de-skew/원근보정 (OpenCV)
  ▼
[검출] DocLayout-YOLO/YOLOv11 fine-tune → 행·셀 bbox
       + 인쇄 격자선 OpenCV Hough/투영으로 열(column) 룰 고정  ← "룰=열, 학습=행"
  │ 셀 crop (원해상도 보존)
  ▼
[인식] 숫자 셀 → PaddleOCR CTC (charset 0-9·천단위·〃 제한) ── 환각 원천 차단
       품목 셀 → korean_PP-OCRv5_mobile_rec (open-vocab)
  ▼
[symbolic IE] 약식 정규화(천원곱·'25×4' 압축·ditto·빈칸=0)
              + 산술검산(수량×단가=공급가, 세액=공급가×10%) 검증·역산 복원
  ▼
[정규화] 품목명 ← 운영DB 386종 사전 + zero-shot entity linking + 신규품목 HITL 큐
  ▼
구조화 JSON {rows[], grand_total} / 검산 실패·저신뢰 → 검수 큐
```

- **검출 주역은 YOLO**(codex/VLM 아님 — VLM grounding 5~21%로 부적합). codex는 검증·부트스트랩만.
- **숫자는 작은 charset CTC**로 환각 차단. **품목명은 open-vocab 인식 + DB 사전 정규화**.
- 비교 측정용 백엔드: A(on-prem Qwen2.5-VL-7B 셀 read), C(TrOCR 흘림체 품목 보조). **macmini = M5 Max/128GB라 on-prem VLM 구동·QLoRA 가능**.

## 5. PoC 경로 (0단계 감사 반영판)

- **0단계 — 데이터 진실 감사** ✅ *(부분 완료)*
  - ✅ 라벨↔운영DB 교차검증(라벨 숫자 99.5~100% 확인), 필드의미 확정, 사전 92% 커버 확인
  - ☐ 남은 것: 약식 표기('25×4'·'〃'·빈칸) 예외율을 **사진 기준**으로 통계 → symbolic 규칙 카탈로그
  - ☐ inv_015(매칭 실패)·inv_030(분해 차이) 사진 확인
  - ☐ **사진↔DB 자동 매칭 파이프라인** 구현(날짜+금액) → 학습 페어 자동 생성기

- **1단계 — B-축소 골격 + 검산 게이트** (단일 경로)
  - 라벨 bbox를 oracle로 사용(검출 노이즈와 인식 품질 분리 측정)
  - 숫자 셀 PaddleOCR CTC raw read → symbolic 정규화 → 산술검산
  - verify: 숫자 string 정확도, 검산 후 (공급가,금액) 행 정합률

- **2단계 — 인식 백엔드 비교** (동일 채점셋·동일 검산 게이트)
  - (CTC) / (A: Qwen2.5-VL-7B 셀 read) / (C: TrOCR 품목명 보조)
  - verify: 백엔드별 셀·행·명세서 완전일치율 → "숫자=?, 품목명=?" 최적 조합

- **3단계 — 데이터 증폭** (실현가능성 입증 후)
  - 운영DB 400건+α를 사진 페어화 → 실데이터 fine-tune (합성보다 우선 — 운영DB가 합성 갭 해소)
  - 보조: 약식↔정규화 합성, AI Hub 한글 손글씨 사전학습, 산술검산 weak supervision
  - verify: held-out 실손글씨에서 행 완전일치 상승

- **4단계 — 운영 판정**
  - 자동확정 비율·HITL 부하로 운영모드 결정 (입력보조 SJMJ-Web 통합 / 무인 / 보류)

## 6. 채점 설계 (0단계 감사 반영)

- **정답지**: 운영DB(공급가·수량·단가·품목명). 라벨은 보조.
- **공급가 기준**: 금액은 `supply`(공급가액)로 채점. `total`(부가세포함)과 혼동 금지.
- **품목명**: **정규화(canonical) 후 일치**로 채점 — 표기변형(가스켓/가스겟)은 동일 취급. "사전 내 vs 사전 밖" 분리 측정.
- **2축 측정**: `raw read 정확도`와 `read+산술검산 후 정확도`를 분리. 검산 게인은 "단일 셀 오류 행"에 한정된 조건부 레버임을 명시(2셀 오류·행 누락 비율 별도 집계).
- **메트릭**: 셀 정확도(필드별) / 행 완전일치율 / 명세서 완전일치율 / 검산 통과율. micro·macro 평균.
- **채점 무게**: 약식 분해 모호성 때문에 (수량,단가) 개별보다 **(공급가, 금액)** 정합에 무게.

## 7. 기술 스택 (전부 on-prem·무료·오픈)

| 단계 | 1순위 | 비고 |
|------|-------|------|
| 검출 | DocLayout-YOLO(DocSynth-300K 사전학습)/YOLOv11 | HF 가중치, KORIE mAP 0.888 |
| 숫자 인식 | PaddleOCR CTC (charset 제한) | 10.5MB, Apache-2.0, 환각 차단 |
| 품목 인식 | korean_PP-OCRv5_mobile_rec | 14MB, open-vocab |
| 보조/비교 | Qwen2.5-VL-7B(MLX) · TrOCR+LoRA | M5 Max 128GB 구동 |
| 사전학습 | AI Hub 한글 손글씨(106,519장·117만) | 무료(내국인) |
| 정규화 | 운영DB 386종 사전 + entity linking | 신규품목 재학습 없이 흡수 |
| 검산 | 자체 symbolic Validator | 산술 항등식 |

## 8. 리스크 (0단계 후 재평가)

**해소됨**
- ✅ 라벨 신뢰도 미검증 → 숫자 99.5~100% 정확 확인
- ✅ 사전 밖 62% → 운영DB 386종으로 92% 커버
- ✅ 합성 domain gap / 실라벨 부족 → 운영DB 400건+α 실정답 + 자동 페어화

**잔여 (관리 필요)**
- 🔶 약식 분해 모호성(inv_030류) → 채점 무게를 공급가/금액에. 예외율 통계 필요
- 🔶 품목명 표기 흔들림(DB조차) → 정규화 레이어 필수, canonical 사전 구축
- 🔶 흘림체 품목명 인식이 진짜 병목 → 1단계 전 baseline 먼저 측정
- 🔶 산술검산 거짓양성(상호보상 오류) → "신뢰도 게이트"로만, pseudo-label은 다중 셀 독립 일치 시에만
- 🔶 검출/인식 fine-tune의 macmini 학습 시간(MPS) — 추론은 여유, 학습 속도는 별도 확인
